## 2. GDB

### 2.1 GDB 用来干什么

GDB 是源码级调试器，用于：

- 设置断点。
- 单步执行。
- 查看变量。
- 查看调用栈。
- 调试多线程。
- 分析 core dump。

编译时通常加：

```bash
g++ -g -O0 main.cpp -o app
```

重点：

- `-g` 生成调试信息。
- `-O0` 更方便调试，优化级别高时源码行和变量可能不直观。
- release 问题也可以带 `-g` 编译，不等于不开优化。

### 2.2 常用命令

| 命令 | 作用 |
|---|---|
| `break main` / `b main` | 设置断点 |
| `run` / `r` | 运行程序 |
| `next` / `n` | 单步越过函数 |
| `step` / `s` | 单步进入函数 |
| `continue` / `c` | 继续运行 |
| `print x` / `p x` | 打印变量 |
| `bt` | 查看调用栈 |
| `frame n` | 切换栈帧 |
| `info threads` | 查看线程 |
| `thread n` | 切换线程 |
| `watch x` | 变量被写时中断 |
| `layout src` | TUI 源码窗口 |

### 2.3 watchpoint

watchpoint 用于观察变量或内存地址何时被修改。

```gdb
watch counter
watch *(int*)0x12345678
```

适合：

- 变量被意外修改。
- 内存踩踏定位。
- 多线程共享变量异常变化。

注意：硬件 watchpoint 数量有限。

### 2.4 coredump

core dump 是进程崩溃时保存的内存现场。

开启 core：

```bash
ulimit -c unlimited
```

分析：

```bash
gdb ./app core
bt
```

重点：

- core 文件用于事后分析。
- 常见崩溃信号：`SIGSEGV`、`SIGABRT`、`SIGFPE`。
- 需要可执行文件和调试符号匹配。

## 3. perf

### 3.1 perf 用来干什么

perf 是 Linux 性能分析工具，用于观察 CPU 事件、热点函数、调用栈、缓存 miss、分支 miss 等。

常见用途：

- 看程序总体性能计数。
- 找 CPU 热点函数。
- 采样调用栈生成火焰图。
- 实时观察系统热点。

### 3.2 常用命令

| 命令 | 作用 |
|---|---|
| `perf stat ./app` | 统计运行期间性能事件 |
| `perf record ./app` | 采样记录性能数据 |
| `perf report` | 交互查看热点报告 |
| `perf top` | 实时查看系统热点 |
| `perf record -g ./app` | 采样调用栈 |

`perf stat` 常见指标：

- cycles
- instructions
- branches
- branch-misses
- cache-references
- cache-misses
- context-switches
- page-faults

### 3.3 火焰图

火焰图展示采样调用栈：

- 横向宽度代表采样占比，不代表时间顺序。
- 越宽说明该路径占用越多 CPU 样本。
- 顶部通常是当前正在消耗 CPU 的函数。

高频结论：

- perf 是采样工具，不是逐条记录所有函数调用。
- 采样频率和符号信息会影响结果。
- CPU 热点不等于所有性能瓶颈，I/O 等待、锁竞争也要结合其他工具。

## 4. strace / ltrace

### 4.1 strace

strace 跟踪系统调用。

常用命令：

```bash
strace ./app
strace -f ./app
strace -p <pid>
strace -T -tt -o trace.log ./app
```

常用选项：

| 选项 | 作用 |
|---|---|
| `-f` | 跟踪 fork/clone 出来的子进程/线程 |
| `-p pid` | attach 到已有进程 |
| `-e trace=open,read,write` | 只跟踪指定 syscall |
| `-T` | 显示 syscall 耗时 |
| `-tt` | 显示精确时间戳 |
| `-o file` | 输出到文件 |

适合排查：

- 程序卡在哪个系统调用。
- 文件打开失败路径。
- 网络连接、读写、权限问题。
- 系统调用返回错误码。

### 4.2 ltrace

ltrace 跟踪动态库函数调用。

```bash
ltrace ./app
```

区别：

| 工具 | 跟踪对象 |
|---|---|
| strace | 系统调用，用户态到内核态接口 |
| ltrace | 动态库函数调用，如 `malloc/printf` |

高频结论：

- `printf` 是库函数，不是系统调用。
- `write` 是系统调用。
- `fread` 可能在用户态缓冲，不一定每次都触发 `read`。

## 5. 系统观测命令

### 5.1 文件与网络

| 命令 | 用途 |
|---|---|
| `lsof -p <pid>` | 查看进程打开的文件、socket、库 |
| `lsof -i :8080` | 查看端口被谁占用 |
| `ss -tulpn` | 查看监听端口和连接 |
| `netstat` | 老牌网络查看工具，很多场景被 `ss` 替代 |

重点：

- `lsof` 能查“谁打开了这个文件/端口”。
- `ss` 通常比 `netstat` 更新、更快。

### 5.2 CPU / 内存 / I/O

| 命令 | 用途 |
|---|---|
| `top` | 实时查看进程 CPU/内存 |
| `htop` | 交互更友好的 top |
| `vmstat 1` | 系统运行队列、内存、swap、I/O、CPU 概览 |
| `iostat -x 1` | 磁盘 I/O 利用率、延迟、吞吐 |
| `mpstat -P ALL 1` | 各 CPU 核心使用情况 |
| `pidstat -p <pid> 1` | 指定进程 CPU/内存/I/O/上下文切换 |

常见判断：

- CPU 高：看 `top/perf`。
- I/O 慢：看 `iostat`。
- 单进程细节：看 `pidstat`。
- 系统整体压力：看 `vmstat`。
- 端口占用：看 `lsof/ss`。

## 6. Sanitizer

### 6.1 AddressSanitizer

ASan 检测内存错误。

```bash
g++ -g -fsanitize=address main.cpp -o app
```

能发现：

- heap buffer overflow
- stack buffer overflow
- use-after-free
- double free
- 部分内存泄漏

### 6.2 ThreadSanitizer

TSan 检测数据竞争。

```bash
g++ -g -fsanitize=thread main.cpp -o app
```

能发现：

- data race
- 锁使用相关问题

注意：

- TSan 运行开销较大。
- 不应和 ASan 在同一个构建里随意混用。

### 6.3 UBSan

UBSan 检测未定义行为。

```bash
g++ -g -fsanitize=undefined main.cpp -o app
```

能发现：

- 有符号整数溢出
- 空指针解引用
- 越界 shift
- 类型转换错误
- 对齐错误

高频结论：

- Sanitizer 是编译期插桩 + 运行期检测。
- 需要重新编译程序。
- 很适合 CI、测试环境，不一定适合生产直接开启。

## 7. valgrind

### 7.1 memcheck

最常用模式：

```bash
valgrind --leak-check=full ./app
```

能发现：

- 内存泄漏。
- 非法读写。
- 使用未初始化值。
- double free。

特点：

- 不需要重新编译插桩，但加 `-g` 更容易定位源码。
- 运行很慢。
- 对多线程性能和时序影响明显。

### 7.2 massif

massif 用于分析堆内存峰值和分配来源。

```bash
valgrind --tool=massif ./app
ms_print massif.out.<pid>
```

适合：内存占用过高、峰值来源分析。

### 7.3 callgrind

callgrind 用于函数调用和指令级性能分析。

```bash
valgrind --tool=callgrind ./app
```

适合：调用关系、函数成本分析。

注意：perf 更接近真实运行性能；callgrind 插桩开销大。

## 9. 高频问答

### Q1：GDB 中 `bt` 是什么？

答：`bt` 是 backtrace，用于查看当前调用栈。

### Q2：watchpoint 和 breakpoint 的区别？

答：breakpoint 在代码位置中断；watchpoint 在变量或内存地址被访问/修改时中断。

### Q3：perf 和 strace 的区别？

答：perf 主要做性能采样和硬件/软件事件统计；strace 跟踪系统调用。

### Q4：strace 的 `-f` 有什么用？

答：跟踪 fork/clone 创建的子进程或线程，否则可能漏掉子进程系统调用。

### Q5：ASan 和 valgrind memcheck 的区别？

答：ASan 需要编译插桩，速度相对快，适合测试；valgrind 不需要插桩但更慢，也能查泄漏和非法访问。

### Q6：TSan 用来查什么？

答：ThreadSanitizer 用来检测多线程 data race。

### Q7：perf 火焰图横向宽度表示什么？

答：表示采样占比，不表示时间先后顺序。

### Q8：`lsof -i :8080` 常用来查什么？

答：查看哪个进程占用了 8080 端口。

## 12. 易错结论速背

- GDB 看崩溃位置和变量；perf 看 CPU 热点；strace 看 syscall。
- `bt` 看调用栈，`info threads` 看线程，`watch` 看变量修改。
- core dump 是崩溃后的现场快照，分析要匹配符号。
- perf 是采样，不是完整追踪每次调用。
- strace 跟系统调用，ltrace 跟动态库函数。
- `printf` 是库函数，`write` 是系统调用。
- ASan 查内存越界/UAF，TSan 查 data race，UBSan 查 UB。
- Sanitizer 需要重新编译插桩。
- valgrind 不需要重新插桩但很慢。
- massif 看堆内存峰值，callgrind 看调用成本。
- 端口占用看 `lsof/ss`，磁盘 I/O 看 `iostat`，单进程资源看 `pidstat`。
