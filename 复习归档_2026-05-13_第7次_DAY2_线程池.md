## 2. 为什么需要线程池

如果每来一个任务就创建一个线程：

- 创建和销毁线程有系统调用和调度成本。
- 线程过多会导致上下文切换增加。
- 资源不可控，可能打爆系统线程数或内存。

线程池思路：

1. 程序启动时创建固定数量工作线程。
2. 外部提交任务到队列。
3. 工作线程从队列取任务执行。
4. 任务执行完后线程不退出，继续等待下一个任务。

适合场景：

- 大量短任务。
- 后台异步任务。
- 服务端请求处理。
- CPU 密集任务控制并发度。

不适合场景：

- 长时间阻塞任务占满所有 worker。
- 每个任务都需要独占线程上下文。
- 实时性极高、调度延迟严格受限的路径。

## 3. 核心数据结构

固定线程池通常包含：

```cpp
class ThreadPool {
private:
    std::vector<std::thread> workers;
    std::queue<std::function<void()>> tasks;
    std::mutex mutex;
    std::condition_variable cv;
    bool stopping = false;
};
```

职责：

| 成员 | 作用 |
|---|---|
| `workers` | 保存工作线程对象，析构时 join |
| `tasks` | 任务队列，生产者提交，消费者执行 |
| `mutex` | 保护任务队列和停止标志 |
| `cv` | 队列为空时阻塞 worker，有任务时唤醒 |
| `stopping` | 表示线程池正在关闭 |

重点：任务队列和 `stopping` 是共享状态，必须用同一把锁保护。

---

## 4. 工作线程循环

典型 worker 逻辑：

```cpp
while (true) {
    std::function<void()> task;

    {
        std::unique_lock<std::mutex> lock(mutex);
        cv.wait(lock, [&] {
            return stopping || !tasks.empty();
        });

        if (stopping && tasks.empty()) {
            return;
        }

        task = std::move(tasks.front());
        tasks.pop();
    }

    task();
}
```

关键点：

- `wait(lock, pred)` 自动处理虚假唤醒。
- 退出条件通常是 `stopping && tasks.empty()`。
- 取出任务后释放锁，再执行任务。
- 不要持锁执行任务，否则其他线程无法提交/取任务，还可能死锁。

## 5. 提交任务

简化版提交：

```cpp
void enqueue(std::function<void()> task) {
    {
        std::lock_guard<std::mutex> lock(mutex);
        if (stopping) {
            throw std::runtime_error("enqueue on stopped ThreadPool");
        }
        tasks.push(std::move(task));
    }
    cv.notify_one();
}
```

重点：

- 修改队列时加锁。
- 提交任务后 `notify_one` 唤醒一个 worker。
- 通常禁止向正在停止的线程池提交任务。
- `notify_one` 可以放在释放锁之后，减少被唤醒线程立刻阻塞在同一把锁上的概率。

带返回值的提交常用 `packaged_task + future`：

```cpp
template <class F>
auto submit(F&& f) -> std::future<decltype(f())> {
    using R = decltype(f());
    auto task = std::make_shared<std::packaged_task<R()>>(std::forward<F>(f));
    std::future<R> result = task->get_future();

    enqueue([task] {
        (*task)();
    });

    return result;
}
```

真实工程中还要支持参数转发、异常传播、返回 `void` 等情况。

## 6. 停止与析构

典型析构流程：

```cpp
~ThreadPool() {
    {
        std::lock_guard<std::mutex> lock(mutex);
        stopping = true;
    }

    cv.notify_all();

    for (auto& worker : workers) {
        if (worker.joinable()) {
            worker.join();
        }
    }
}
```

重点：

- 设置 `stopping = true` 后要 `notify_all`，否则空队列中睡眠的 worker 可能永远醒不过来。
- worker 退出前一般会把队列中已有任务处理完。
- 析构时必须 join 所有 joinable 线程。
- 析构函数不能在 worker 自己内部调用并 join 自己，否则会死锁或抛异常。

两种停止语义：

| 语义 | 含义 |
|---|---|
| drain shutdown | 不接收新任务，但执行完队列已有任务再退出 |
| immediate shutdown | 尽快退出，丢弃或取消队列中未执行任务 |

笔试默认更常见的是 drain shutdown。

## 7. 最小线程池骨架

```cpp
#include <condition_variable>
#include <functional>
#include <mutex>
#include <queue>
#include <stdexcept>
#include <thread>
#include <vector>

class ThreadPool {
public:
    explicit ThreadPool(size_t thread_count) {
        for (size_t index = 0; index < thread_count; ++index) {
            workers.emplace_back([this] {
                worker_loop();
            });
        }
    }

    ~ThreadPool() {
        {
            std::lock_guard<std::mutex> lock(mutex);
            stopping = true;
        }

        cv.notify_all();

        for (auto& worker : workers) {
            if (worker.joinable()) {
                worker.join();
            }
        }
    }

    void enqueue(std::function<void()> task) {
        {
            std::lock_guard<std::mutex> lock(mutex);
            if (stopping) {
                throw std::runtime_error("enqueue on stopped ThreadPool");
            }
            tasks.push(std::move(task));
        }
        cv.notify_one();
    }

private:
    void worker_loop() {
        while (true) {
            std::function<void()> task;

            {
                std::unique_lock<std::mutex> lock(mutex);
                cv.wait(lock, [this] {
                    return stopping || !tasks.empty();
                });

                if (stopping && tasks.empty()) {
                    return;
                }

                task = std::move(tasks.front());
                tasks.pop();
            }

            task();
        }
    }

    std::vector<std::thread> workers;
    std::queue<std::function<void()>> tasks;
    std::mutex mutex;
    std::condition_variable cv;
    bool stopping = false;
};
```

这个骨架能说明核心思想，但还不是工业级实现。

## 8. 工程级线程池还会考虑什么

- 任务返回值和异常传播：`future / packaged_task`。
- 队列容量限制：防止生产速度远超消费速度。
- 拒绝策略：阻塞提交、抛异常、返回 false、丢弃旧任务。
- 动态扩缩容：根据负载调整线程数。
- 任务优先级：优先队列。
- work stealing：每个 worker 一个本地队列，空闲时偷任务。
- CPU 亲和性：绑定核心，减少迁移。
- NUMA 感知：让任务和数据尽量在同一 NUMA 节点。
- 优雅关闭：drain、cancel、timeout。

## 9. 常见坑

### 9.1 持锁执行任务

错误做法：

```cpp
std::lock_guard<std::mutex> lock(mutex);
auto task = tasks.front();
tasks.pop();
task(); // 错：持锁执行未知任务
```

风险：

- 阻塞其他 worker 取任务。
- 阻塞提交线程 enqueue。
- 任务内部如果再次提交任务或等待线程池任务，可能死锁。

### 9.2 忘记 `notify_all`

析构只设置 `stopping = true`，但不通知等待线程，worker 可能永远睡眠，析构 join 卡死。

### 9.3 条件变量用 `if`

```cpp
if (tasks.empty()) {
    cv.wait(lock);
}
```

错因：虚假唤醒或多个 worker 抢任务后，当前线程醒来时队列可能仍为空。

### 9.4 任务异常导致线程退出

如果 `task()` 抛异常且 worker 没捕获，线程可能直接终止，线程池容量悄悄减少。

工程里常见处理：

- 用 `packaged_task` 把异常传给 `future`。
- worker 捕获异常并记录日志，避免线程无声死亡。

### 9.5 队列无限增长

如果提交速度大于执行速度，无界队列会持续吃内存。

解决：

- 有界队列。
- backpressure。
- 拒绝策略。

## 10. 高频问答

### Q1：线程池解决的核心问题是什么？

答：复用固定数量线程，减少频繁创建/销毁线程的开销，并限制并发度，避免线程数量失控。

### Q2：线程池的核心组件有哪些？

答：工作线程数组、任务队列、互斥锁、条件变量、停止标志。

### Q3：worker 等待条件为什么是 `stopping || !tasks.empty()`？

答：有任务时要醒来执行；停止时即使没有任务也要醒来退出。

### Q4：为什么取出任务后要先释放锁再执行？

答：避免长时间持锁阻塞其他线程提交/取任务，也避免任务内部重入线程池导致死锁。

### Q5：析构时为什么要 `notify_all`？

答：一些 worker 可能正在空队列上睡眠。只设置停止标志不通知，它们不会醒来，join 会卡住。

### Q6：线程池停止时一定要丢弃未执行任务吗？

答：不一定。常见有 drain shutdown 和 immediate shutdown 两种语义，笔试默认通常执行完已有任务再退出。

### Q7：线程池任务抛异常怎么办？

答：不能让异常无声杀死 worker。可用 `packaged_task/future` 传递异常，或 worker 捕获并记录。

### Q8：线程数是不是越多越好？

答：不是。CPU 密集任务一般接近 CPU 核心数；I/O 密集可适当更多。线程过多会增加上下文切换和内存开销。

## 11. 选择题训练

### 题 1

线程池的主要目的之一是：

A. 每个任务都创建新线程  
B. 复用线程，减少创建销毁开销  
C. 禁止并发执行  
D. 替代所有锁

答案：B

解析：线程池通过复用 worker 控制并发度和线程生命周期成本。

### 题 2

固定线程池通常不需要的组件是：

A. 工作线程数组  
B. 任务队列  
C. 条件变量  
D. GPU 纹理单元

答案：D

解析：线程池核心是 worker、queue、mutex、condition_variable、stop flag。

### 题 3

worker 从任务队列取出任务后，正确做法通常是：

A. 持锁执行任务  
B. 释放锁后执行任务  
C. 删除所有 worker  
D. 立即退出进程

答案：B

解析：持锁执行未知任务容易阻塞其他线程甚至造成死锁。

### 题 4

线程池析构时设置停止标志后通常要：

A. `notify_all` 唤醒 worker  
B. 忽略所有线程  
C. 调用 `fork`  
D. 清空标准输出

答案：A

解析：空队列上等待的 worker 需要被唤醒才能检查停止条件并退出。

### 题 5

条件变量等待任务时应使用：

A. `if` 只判断一次  
B. `while` 或 `wait(lock, pred)`  
C. 不加锁直接访问队列  
D. 忙等空循环

答案：B

解析：必须处理虚假唤醒和竞争，推荐 `wait(lock, pred)`。

### 题 6

drain shutdown 的含义是：

A. 立即杀死进程  
B. 执行完队列已有任务后退出  
C. 永远不退出  
D. 丢弃所有任务且不通知线程

答案：B

解析：drain 表示排空已有任务，不再接收新任务。

### 题 7

无界任务队列的风险是：

A. 永远不会占内存  
B. 提交速度过快时内存持续增长  
C. 自动变成 lock-free  
D. 无法存放任务

答案：B

解析：生产速度大于消费速度时，无界队列会堆积任务。

### 题 8

带返回值的任务提交常用：

A. `std::packaged_task` 和 `std::future`  
B. `printf` 和 `scanf`  
C. `goto`  
D. `errno`

答案：A

解析：`packaged_task/future` 可把任务结果和异常传回提交者。

### 题 9

CPU 密集型线程池线程数通常接近：

A. CPU 核心数  
B. 文件数量  
C. 网络端口数  
D. 字符串长度

答案：A

解析：CPU 密集任务主要消耗 CPU，线程过多会增加上下文切换。

### 题 10

任务在 worker 中抛异常且未捕获，可能导致：

A. worker 线程终止  
B. 自动重试所有任务  
C. 编译器报错  
D. 队列变成有界队列

答案：A

解析：未捕获异常会终止当前线程，线程池容量可能减少。

## 13. 易错结论速背

- 线程池 = 固定 worker + 任务队列 + mutex + condition_variable + stop flag。
- worker 等待条件通常是 `stopping || !tasks.empty()`。
- 退出条件常见是 `stopping && tasks.empty()`。
- 条件变量必须防虚假唤醒，用 `while` 或 `wait(lock, pred)`。
- 取任务时持锁，执行任务时不持锁。
- 析构先设置 stop，再 `notify_all`，最后 join workers。
- joinable 的 `std::thread` 析构会 `std::terminate()`。
- drain shutdown 是执行完已有任务再退出。
- 任务异常要通过 `future` 传递或在线程内捕获，不能让 worker 无声死亡。
- 无界队列有内存风险，工程中要考虑限流和拒绝策略。
- 线程数不是越多越好，CPU 密集接近核心数，I/O 密集可适当更多。
