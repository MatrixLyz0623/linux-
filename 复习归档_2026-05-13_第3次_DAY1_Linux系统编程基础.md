## 2. 文件 I/O 速记

### 2.1 文件描述符 fd

Linux 中进程通过文件描述符访问文件、管道、socket、设备等内核对象。

默认 3 个 fd：

| fd | 含义 |
|---|---|
| 0 | 标准输入 stdin |
| 1 | 标准输出 stdout |
| 2 | 标准错误 stderr |

重点：

- fd 是进程内的小整数索引。
- fd 指向内核维护的打开文件表项。
- `open` 成功返回 fd，失败返回 `-1` 并设置 `errno`。

### 2.2 `open`

```c
#include <fcntl.h>
#include <sys/stat.h>

int fd = open("data.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
```

常见 flags：

| flag | 含义 | 常考点 |
|---|---|---|
| `O_RDONLY` | 只读 | 只读打开 |
| `O_WRONLY` | 只写 | 只写打开 |
| `O_RDWR` | 读写 | 读写打开 |
| `O_CREAT` | 不存在则创建 | 需要第三个参数 mode |
| `O_TRUNC` | 打开时截断为 0 | 常用于覆盖写 |
| `O_APPEND` | 每次写追加到文件末尾 | 内核保证追加位置原子更新 |
| `O_NONBLOCK` | 非阻塞 I/O | pipe/socket/设备常见 |
| `O_SYNC` | 同步写 | 写入等待数据落盘，慢 |
| `O_DIRECT` | 尽量绕过页缓存 | 有对齐要求，不等于绝对直达硬盘 |

权限 mode：

```c
0644 // owner 可读写，group/other 只读
```

最终权限还会受到 `umask` 影响。

### 2.3 `read` / `write`

```c
ssize_t n = read(fd, buffer, sizeof(buffer));
ssize_t m = write(fd, buffer, n);
```

返回值速记：

| 调用 | 返回值 | 含义 |
|---|---|---|
| `read` | `> 0` | 实际读到的字节数 |
| `read` | `0` | EOF，文件读完或对端关闭 |
| `read` | `-1` | 出错，查看 `errno` |
| `write` | `>= 0` | 实际写入字节数 |
| `write` | `-1` | 出错，查看 `errno` |

重点：

- `read/write` 不保证一次读满或写满请求字节数。
- socket、pipe、非阻塞 fd 上尤其容易出现短读短写。
- 工程里写可靠 I/O 时，要循环处理剩余字节。

### 2.4 `close`

```c
close(fd);
```

重点：

- 释放进程持有的 fd。
- fd 关闭后可能被后续 `open` 复用。
- 关闭后继续使用旧 fd 是错误行为。

### 2.5 `lseek`

```c
off_t pos = lseek(fd, 0, SEEK_END);
```

常见 whence：

| whence | 基准位置 |
|---|---|
| `SEEK_SET` | 文件开头 |
| `SEEK_CUR` | 当前位置 |
| `SEEK_END` | 文件末尾 |

重点：

- `lseek` 修改文件偏移，不读写数据。
- 管道、socket 通常不能 `lseek`。
- 可以用 `lseek(fd, 0, SEEK_CUR)` 获取当前偏移。

### 2.6 `fcntl`

`fcntl` 是文件描述符控制接口，功能很多。

高频用途：

- 获取/设置 fd flags：`F_GETFL / F_SETFL`
- 设置非阻塞：给原 flags 加 `O_NONBLOCK`
- 文件锁：`F_SETLK / F_SETLKW`
- 设置 close-on-exec：`FD_CLOEXEC`

```c
int flags = fcntl(fd, F_GETFL, 0);
fcntl(fd, F_SETFL, flags | O_NONBLOCK);
```

## 3. C 库 vs syscall

### 3.1 `fread/fwrite` vs `read/write`

| 对比 | `fread/fwrite` | `read/write` |
|---|---|---|
| 层级 | C 标准库 stdio | Linux 系统调用 |
| 操作对象 | `FILE*` | fd |
| 缓冲 | 用户态 stdio 缓冲 | 直接进入内核 |
| 可移植性 | C 标准库，跨平台更好 | POSIX/Linux 风格 |
| 常见场景 | 普通文件读写、格式化 I/O | 系统编程、socket、pipe、设备 |

重点：

- `FILE*` 内部通常包装了一个 fd。
- `fread/fwrite` 可能先读写 stdio 用户态缓冲区，不一定立刻触发系统调用。
- `read/write` 每次调用都直接进入内核，但可能被页缓存接住，不等于直接访问磁盘。

### 3.2 `fflush` vs `fsync`

| 调用 | 作用范围 | 关键点 |
|---|---|---|
| `fflush(FILE*)` | 刷新 C 库用户态缓冲到内核 | 数据可能还在内核页缓存 |
| `fsync(fd)` | 要求内核把文件数据和元数据刷到磁盘 | 更接近持久化，成本高 |
| `fdatasync(fd)` | 刷数据，尽量少刷元数据 | 比 `fsync` 略轻 |

高频陷阱：

- `fflush` 不等于落盘。
- `write` 返回成功不等于数据已经物理写入磁盘。
- 要求崩溃后尽量不丢数据，要考虑 `fsync`。

### 3.3 缓冲类型

stdio 常见缓冲：

- 全缓冲：普通文件常见，缓冲区满才刷新。
- 行缓冲：终端输出常见，遇到换行刷新。
- 无缓冲：`stderr` 常见。

## 4. 进程创建与回收

### 4.1 `fork`

```c
pid_t pid = fork();

if (pid == 0) {
    // 子进程
} else if (pid > 0) {
    // 父进程，pid 是子进程 pid
} else {
    // fork 失败
}
```

返回值：

| 返回值 | 所在进程 | 含义 |
|---|---|---|
| `0` | 子进程 | 当前是子进程 |
| `> 0` | 父进程 | 返回子进程 pid |
| `-1` | 父进程 | 创建失败 |

重点：

- `fork` 后父子进程从同一行之后继续执行。
- 父子进程有独立地址空间。
- 父子进程会继承打开的 fd，文件偏移等内核打开文件状态可能共享。

### 4.2 写时复制 COW

`fork` 后不会立刻复制全部物理内存。

机制：

1. 父子进程先共享物理页。
2. 页表标记为只读/COW。
3. 某一方写入时触发缺页异常。
4. 内核复制该页，写入方获得自己的副本。

高频结论：

- `fork` 看起来复制地址空间，实际常用 COW 降低成本。
- 父子进程变量互不影响，因为写入时会复制对应页。

### 4.3 `exec`

`exec` 系列用新程序替换当前进程映像。

```c
execl("/bin/ls", "ls", "-l", NULL);
```

重点：

- `exec` 成功后不会返回。
- 进程 pid 通常不变。
- 原来的代码段、数据段、堆栈被新程序替换。
- 未设置 close-on-exec 的 fd 默认会保留到新程序中。

### 4.4 `wait` / `waitpid`

```c
int status;
pid_t child = waitpid(pid, &status, 0);
```

作用：

- 等待子进程状态变化。
- 回收子进程退出状态。
- 防止僵尸进程长期存在。

常用宏：

| 宏 | 含义 |
|---|---|
| `WIFEXITED(status)` | 是否正常退出 |
| `WEXITSTATUS(status)` | 获取退出码 |
| `WIFSIGNALED(status)` | 是否被信号杀死 |
| `WTERMSIG(status)` | 获取终止信号 |

`waitpid` 优势：

- 可以等待指定 pid。
- 支持 `WNOHANG` 非阻塞轮询。

### 4.5 僵尸进程与孤儿进程

僵尸进程：

- 子进程已经退出。
- 父进程还没有 `wait/waitpid` 回收。
- 内核保留少量退出状态信息。
- 状态常见为 `Z`。

孤儿进程：

- 父进程先退出，子进程还在运行。
- 子进程会被 init/systemd 等进程接管。
- 孤儿进程不是错误，通常会被新父进程回收。

### 4.6 `SIGCHLD`

子进程退出或停止时，内核会给父进程发送 `SIGCHLD`。

常见处理方式：

- 父进程捕获 `SIGCHLD`，在 handler 或主循环中 `waitpid` 回收。
- 使用 `waitpid(-1, &status, WNOHANG)` 循环回收多个已退出子进程。

重点：

- 信号可能合并，不能假设一次 `SIGCHLD` 只对应一个子进程。
- signal handler 中只能调用异步信号安全函数，复杂逻辑应放到主循环。

## 5. 高频问答

### Q1：`read` 返回 0 表示什么？

答：对普通文件通常表示 EOF；对 pipe/socket 常表示对端关闭写端或连接关闭。

### Q2：`write` 成功返回是否说明数据已经落盘？

答：不一定。通常只说明数据交给内核，可能还在页缓存中。需要持久化时要考虑 `fsync`。

### Q3：`fflush` 和 `fsync` 的区别是什么？

答：`fflush` 刷 C 库用户态缓冲到内核；`fsync` 要求内核把文件数据同步到磁盘。

### Q4：`fork` 后父子进程变量会互相影响吗？

答：不会。父子进程地址空间独立，底层通过写时复制优化，写入时会复制对应物理页。

### Q5：`exec` 成功后会返回吗？

答：不会。成功后当前进程映像被新程序替换；只有失败才返回 `-1`。

### Q6：僵尸进程怎么产生？怎么解决？

答：子进程退出后父进程没有回收就会变成僵尸。父进程应调用 `wait/waitpid`，也可通过处理 `SIGCHLD` 及时回收。

## 6. 选择题训练

### 题 1

`open` 成功时返回的是：

A. `FILE*`  
B. 文件描述符 fd  
C. 文件路径字符串  
D. inode 指针

答案：B

解析：`open` 是系统调用接口，成功返回 fd；`fopen` 返回 `FILE*`。

### 题 2

使用 `O_CREAT` 创建文件时，`open` 通常还需要提供：

A. 文件大小  
B. mode 权限参数  
C. 进程 pid  
D. inode 编号

答案：B

解析：`O_CREAT` 场景下需要第三个参数 mode，如 `0644`。

### 题 3

`read` 返回 0 通常表示：

A. 出错  
B. EOF 或对端关闭  
C. 成功读到 0 字节但还有大量数据  
D. 文件描述符自动关闭

答案：B

解析：`read` 返回 0 是读到结束的典型信号。

### 题 4

下列哪个调用用于改变文件偏移？

A. `lseek`  
B. `fcntl`  
C. `fsync`  
D. `fork`

答案：A

解析：`lseek` 修改当前文件偏移。

### 题 5

`fflush(fp)` 的主要作用是：

A. 强制把数据写入物理磁盘  
B. 刷新 C 库用户态缓冲  
C. 关闭 fd  
D. 创建新进程

答案：B

解析：`fflush` 只负责 stdio 缓冲，不等于落盘。

### 题 6

`fork` 在子进程中的返回值是：

A. `0`  
B. 子进程 pid  
C. 父进程 pid  
D. `-2`

答案：A

解析：`fork` 子进程返回 0，父进程返回子进程 pid。

### 题 7

`exec` 成功后：

A. 返回 0 继续执行原程序  
B. 当前进程映像被新程序替换  
C. 一定创建新进程  
D. 父进程被杀死

答案：B

解析：`exec` 不创建新进程，而是替换当前进程映像。

### 题 8

子进程退出后父进程长期不调用 `wait`，子进程可能成为：

A. 僵尸进程  
B. 孤儿进程  
C. 内核线程  
D. 守护进程

答案：A

解析：退出未回收的子进程是僵尸进程。

### 题 9

父进程退出而子进程仍在运行，这个子进程称为：

A. 僵尸进程  
B. 孤儿进程  
C. 死锁进程  
D. 交换进程

答案：B

解析：父进程先退出，子进程会成为孤儿并被 init/systemd 接管。

### 题 10

`fork` 使用写时复制的主要目的是：

A. 让父子进程共享所有变量修改  
B. 降低立即复制整个地址空间的成本  
C. 禁止子进程写内存  
D. 替代 `exec`

答案：B

解析：COW 让物理页按需复制，降低 `fork` 成本。

## 8. 易错结论速背

- `open` 返回 fd，`fopen` 返回 `FILE*`。
- `read/write` 直接用 fd；`fread/fwrite` 用 `FILE*` 并经过 stdio 缓冲。
- `read` 返回 0 是 EOF 或对端关闭。
- `write` 可能短写，成功不等于落盘。
- `fflush` 刷用户态缓冲，`fsync` 刷内核页缓存到磁盘。
- `O_CREAT` 创建文件时要给 mode，最终权限受 `umask` 影响。
- `lseek` 改偏移，不读写数据；pipe/socket 通常不可 seek。
- `fork` 父进程返回子 pid，子进程返回 0。
- `fork` 后父子进程地址空间独立，底层靠 COW 优化。
- `exec` 成功不返回，不创建新进程，只替换当前进程映像。
- 子进程退出未回收是僵尸；父进程先退出、子进程仍运行是孤儿。
- `SIGCHLD` 可能合并，回收子进程要循环 `waitpid`。
