## 2. 为什么需要内存序

多线程下，问题不只是“操作是不是原子”，还包括：

- 一个线程写入的数据，另一个线程什么时候可见。
- 编译器和 CPU 是否能重排序指令。
- 原子变量是否能建立跨线程同步关系。

最短记忆：

- 原子性：这个操作不会被撕裂。
- 可见性：一个线程写入，另一个线程能不能看到。
- 有序性：操作之间的先后关系会不会被重排。

`std::atomic` 解决的是原子操作和内存序问题，但不是自动解决所有业务不变量。

## 3. C++ 内存序速记

### 3.1 总览表

| 内存序 | 含义 | 常见用途 |
|---|---|---|
| `memory_order_relaxed` | 只保证原子性，不保证同步顺序 | 计数器、统计值 |
| `memory_order_acquire` | acquire 读，禁止后续读写跑到它前面 | 获取发布的数据 |
| `memory_order_release` | release 写，禁止之前读写跑到它后面 | 发布数据给其他线程 |
| `memory_order_acq_rel` | 读改写同时具备 acquire + release | `fetch_add`、CAS 等 RMW |
| `memory_order_seq_cst` | 最强顺序，一致的全局顺序 | 默认选项，最容易推理 |

一句话：

- `relaxed`：只管这个原子变量本身。
- `release`：我之前写好的东西，现在发布出去。
- `acquire`：我看到发布信号后，也能看到发布前写好的东西。
- `seq_cst`：所有线程对原子操作看到一个统一总顺序。

### 3.2 relaxed

```cpp
std::atomic<int> counter{0};
counter.fetch_add(1, std::memory_order_relaxed);
```

适合：

- 统计计数。
- 性能指标。
- 不依赖该原子变量同步其他数据。

重点：

- `relaxed` 仍然是原子的。
- `relaxed` 不建立 happens-before。
- 不能用 `relaxed` flag 来安全发布普通数据。

### 3.3 release / acquire

经典发布-订阅模型：

```cpp
int data = 0;
std::atomic<bool> ready{false};

// thread 1
data = 42;
ready.store(true, std::memory_order_release);

// thread 2
if (ready.load(std::memory_order_acquire)) {
    // 这里能看到 data == 42
}
```

重点：

- release store 之前的写入，对 acquire load 之后的读取可见。
- acquire 必须读到对应 release 写入的值，才建立同步关系。
- 这是无锁发布数据最常考模型。

### 3.4 acq_rel

用于读-改-写原子操作。

```cpp
flag.test_and_set(std::memory_order_acquire);
counter.fetch_add(1, std::memory_order_acq_rel);
```

常见于：

- CAS 成功时既读取旧值，又发布新值。
- 自旋锁加解锁的组合逻辑。
- 引用计数等 RMW 操作。

### 3.5 seq_cst

`seq_cst` 是默认内存序，也是最强内存序。

```cpp
std::atomic<int> x{0};
x.store(1); // 默认 seq_cst
```

重点：

- 最容易推理。
- 性能可能比弱内存序差。
- 笔试中不确定时，默认 `seq_cst` 是安全但可能不最优的答案。

## 4. happens-before

happens-before 表示一个操作的结果对另一个操作可见，且顺序受约束。

常见建立方式：

- 同一线程内，前面的语句 sequenced-before 后面的语句。
- mutex unlock happens-before 后续成功 lock 同一个 mutex。
- release store 与读到该值的 acquire load 建立同步。
- `std::thread` 创建前的操作对新线程入口可见。
- 线程结束 happens-before 成功 `join` 返回。

选择题关键词：

- 没有 happens-before 的并发普通读写，可能是 data race。
- data race 在 C++ 中是未定义行为。
- 原子操作本身不会产生 data race，但不代表业务逻辑一定正确。

## 5. CAS 与无锁编程

### 5.1 CAS 是什么

CAS：Compare-And-Swap，比较并交换。

语义：

```text
如果 当前值 == 期望值 expected：
    写入 desired，返回成功
否则：
    把当前值写回 expected，返回失败
```

C++ API：

```cpp
std::atomic<int> value{0};
int expected = 0;
bool ok = value.compare_exchange_strong(expected, 1);
```

重点：

- CAS 是许多无锁数据结构的基础。
- CAS 失败后，`expected` 会被更新为当前实际值。
- 常见写法是放在循环里重试。

### 5.2 weak vs strong

| API | 特点 | 常见用途 |
|---|---|---|
| `compare_exchange_weak` | 允许伪失败 | 循环中更常用 |
| `compare_exchange_strong` | 不应伪失败 | 单次判断或不想处理伪失败 |

伪失败：明明当前值等于 expected，也可能返回失败。某些架构上这样实现更高效。

### 5.3 CAS 循环例子

```cpp
std::atomic<int> counter{0};

void add_one() {
    int old = counter.load();
    while (!counter.compare_exchange_weak(old, old + 1)) {
        // 失败时 old 已被更新为 counter 当前值
    }
}
```

重点：

- 循环中不需要手动重新 load，失败时 `old` 已更新。
- 真实代码里普通自增可直接用 `fetch_add`，这个例子主要帮助理解 CAS。

### 5.4 lock-free / wait-free / obstruction-free

| 概念 | 含义 |
|---|---|
| lock-free | 系统整体总有某个线程能推进 |
| wait-free | 每个线程都能在有限步内完成 |
| obstruction-free | 没有竞争时能完成 |

高频结论：

- lock-free 不等于没有等待。
- lock-free 不等于一定比 mutex 快。
- 无锁代码通常更难验证，容易出现 ABA、内存回收问题。

## 6. ABA 问题

### 6.1 ABA 是什么

一个线程看到变量从 A 变成 B，又变回 A，以为“没变过”，但实际状态已经发生变化。

典型场景：无锁栈。

```text
线程 T1 读到 top = A
线程 T2 pop A，pop B，又 push A
线程 T1 CAS(top, A, next) 成功
但此时 A 的 next 关系可能已经不是原来的结构
```

问题本质：CAS 只比较值是否相等，不知道中间有没有变化历史。

### 6.2 解决思路

常见方法：

- 指针 + 版本号/tag，每次修改版本号递增。
- hazard pointer：声明自己正在访问的节点，避免被提前回收。
- epoch-based reclamation：按阶段延迟回收内存。
- 使用 GC 或引用计数管理节点生命周期。

选择题记忆：ABA 常见于无锁栈/队列，根因是“值相同但历史不同”。

## 7. 无锁队列思路

经典无锁队列通常维护：

- `head`：出队位置。
- `tail`：入队位置。
- 节点链表或环形缓冲区。

基本思路：

1. 入队线程准备新节点。
2. 用 CAS 尝试把新节点挂到 tail 后面。
3. 成功后推进 tail。
4. 出队线程用 CAS 推进 head。

高频难点：

- ABA 问题。
- 节点内存何时安全释放。
- 多生产者多消费者下 head/tail 竞争。
- cache line 竞争和伪共享。

笔试一般不要求手写完整无锁队列，但会考 CAS、ABA、原子变量和内存序。

## 8. 常见并发 bug

### 8.1 死锁 deadlock

死锁四个必要条件：

1. 互斥条件：资源一次只能被一个线程占有。
2. 占有并等待：持有资源同时等待其他资源。
3. 不可抢占：资源不能被强制剥夺。
4. 循环等待：线程之间形成环形等待关系。

解决思路：

- 固定加锁顺序。
- 一次性申请多个锁，如 `std::lock`。
- 使用超时锁 `try_lock_for`。
- 减小锁粒度和持锁时间。
- 避免持锁调用外部未知代码。

### 8.2 活锁 livelock

活锁：线程都在运行、都在让步或重试，但没有实际进展。

例子：两个线程不断互相让路，结果谁也过不去。

区别：

- 死锁：线程卡住不动。
- 活锁：线程一直动，但没有进展。

解决：随机退避、指数退避、打破对称性。

### 8.3 优先级反转

高优先级线程等待低优先级线程持有的锁，而中优先级线程不断抢占低优先级线程，导致高优先级线程迟迟不能运行。

解决思路：

- 优先级继承。
- 优先级天花板。
- 减少高优先级路径上的锁竞争。

### 8.4 伪共享 false sharing

多个线程修改不同变量，但这些变量落在同一个 cache line，导致缓存一致性协议频繁失效。

```cpp
struct Counter {
    alignas(64) std::atomic<long> value;
};
```

重点：

- 伪共享不是数据竞争，变量可能各自独立。
- 伪共享主要造成性能下降。
- 常通过 cache line 对齐和 padding 解决。
- 常见 cache line 大小是 64 字节，但具体平台可能不同。

## 9. 高频问答

### Q1：`memory_order_relaxed` 是不是非原子？

答：不是。`relaxed` 仍保证原子性，只是不建立跨线程同步顺序。

### Q2：release/acquire 用来解决什么？

答：用于发布-订阅。release 线程先写普通数据再发布 flag，acquire 线程读到 flag 后能看到发布前的数据。

### Q3：什么是 happens-before？

答：表示一个操作在并发语义上先于另一个操作，前者结果对后者可见。没有 happens-before 的普通并发读写可能形成 data race。

### Q4：CAS 失败时 `expected` 会怎样？

答：会被更新为当前实际值，所以 CAS 循环里通常直接复用 `expected`。

### Q5：ABA 问题的本质是什么？

答：值从 A 变成 B 又变回 A，CAS 只看到当前值仍是 A，却不知道中间发生过变化。

### Q6：lock-free 是否一定比 mutex 快？

答：不一定。无锁代码在高竞争、复杂内存回收、cache 抖动下可能更慢，也更难写对。

### Q7：死锁四个必要条件是什么？

答：互斥、占有并等待、不可抢占、循环等待。

### Q8：伪共享是不是 data race？

答：不是。伪共享可以发生在不同线程修改不同变量时，主要问题是 cache line 失效导致性能下降。

## 10. 选择题训练

### 题 1

`memory_order_relaxed` 保证的是：

A. 原子性  
B. 全局顺序一致  
C. 自动加 mutex  
D. 禁止所有重排序

答案：A

解析：`relaxed` 只保证原子操作本身，不建立同步关系。

### 题 2

发布-订阅模型中常用的内存序组合是：

A. release / acquire  
B. relaxed / relaxed  
C. const / volatile  
D. malloc / free

答案：A

解析：release 发布数据，acquire 获取发布结果并看到之前写入。

### 题 3

C++ 中没有 happens-before 的普通并发读写可能导致：

A. data race，未定义行为  
B. 自动序列化  
C. 自动变成 atomic  
D. 编译器强制报错

答案：A

解析：C++ 数据竞争是未定义行为。

### 题 4

CAS 的全称通常是：

A. Copy And Store  
B. Compare And Swap  
C. Cache And Sync  
D. Create And Sleep

答案：B

解析：CAS 表示比较并交换。

### 题 5

`compare_exchange_weak` 的特点是：

A. 可能伪失败  
B. 不能用于循环  
C. 一定比 strong 正确性差  
D. 不是原子操作

答案：A

解析：weak 允许伪失败，常放在循环中使用。

### 题 6

ABA 问题的典型场景是：

A. 无锁栈/队列  
B. 单线程局部变量  
C. 只读常量  
D. 编译期模板

答案：A

解析：ABA 常见于 CAS 驱动的无锁数据结构。

### 题 7

下列哪项不是死锁四个必要条件之一？

A. 互斥  
B. 占有并等待  
C. 不可抢占  
D. 栈溢出

答案：D

解析：死锁四条件是互斥、占有并等待、不可抢占、循环等待。

### 题 8

活锁的特点是：

A. 线程完全不运行  
B. 线程一直运行但没有进展  
C. 一定发生段错误  
D. 一定是内存泄漏

答案：B

解析：活锁不是卡死，而是不停动作却无法推进。

### 题 9

优先级反转中，高优先级线程通常在等待：

A. 低优先级线程持有的锁  
B. 自己的栈  
C. 编译器优化  
D. 字符串常量

答案：A

解析：高优先级任务被低优先级持锁任务间接阻塞。

### 题 10

伪共享主要导致：

A. 编译失败  
B. 性能下降  
C. 文件损坏  
D. 进程一定退出

答案：B

解析：多个变量位于同一 cache line，被不同线程频繁修改，引发 cache line 抖动。

## 12. 易错结论速背

- atomic 解决原子性和内存序，不自动保证复杂业务不变量。
- `relaxed` 是原子的，但不建立同步。
- `release/acquire` 是发布-订阅经典组合。
- `seq_cst` 最强、默认、最好推理，但可能性能成本更高。
- happens-before 决定可见性和顺序；没有它的普通并发读写可能 UB。
- CAS 失败会更新 `expected`。
- `compare_exchange_weak` 可能伪失败，常用于循环。
- ABA 本质是“值看起来没变，历史已经变了”。
- lock-free 不等于 wait-free，也不等于一定更快。
- 死锁四条件：互斥、占有并等待、不可抢占、循环等待。
- 活锁是一直动但没有进展。
- 优先级反转：高优先级等低优先级持有的锁。
- 伪共享不是 data race，主要是性能问题，用 padding/alignas 缓解。
