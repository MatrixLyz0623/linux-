## 2. 信号 Signal

### 2.1 信号是什么

信号是 Linux 中一种异步通知机制，用于通知进程发生了某个事件。

常见信号：

| 信号 | 含义 | 默认行为 |
|---|---|---|
| `SIGINT` | 终端 Ctrl+C | 终止进程 |
| `SIGTERM` | 请求终止 | 终止进程 |
| `SIGKILL` | 强制杀死 | 终止进程，不能捕获/忽略 |
| `SIGSTOP` | 暂停进程 | 暂停，不能捕获/忽略 |
| `SIGSEGV` | 段错误 | 终止并可能生成 core |
| `SIGCHLD` | 子进程状态变化 | 默认忽略 |
| `SIGPIPE` | 向已关闭管道/socket 写 | 终止进程 |
| `SIGALRM` | 定时器到期 | 终止进程 |

重点：

- 信号是异步的，可能在程序执行任意位置打断当前流程。
- `SIGKILL` 和 `SIGSTOP` 不能被捕获、阻塞或忽略。
- 普通信号可能合并，不保证排队多次。
- 实时信号支持排队，但选择题一般先记普通信号可能丢次数。

### 2.2 `signal` vs `sigaction`

```c
#include <signal.h>

void handler(int signo) {}

struct sigaction sa;
sa.sa_handler = handler;
sigemptyset(&sa.sa_mask);
sa.sa_flags = 0;
sigaction(SIGINT, &sa, NULL);
```

| 对比 | `signal` | `sigaction` |
|---|---|---|
| 标准语义 | 历史接口，不同系统语义可能不同 | POSIX 推荐接口 |
| 可控性 | 简单 | 可设 mask、flags、更多行为 |
| 推荐度 | 简单 demo 可用 | 工程中优先使用 |

常见 `sigaction` flags：

- `SA_RESTART`：被信号打断的某些系统调用自动重启。
- `SA_SIGINFO`：使用带更多信息的 handler。
- `SA_NOCLDWAIT`：子进程退出不产生僵尸。

### 2.3 可重入函数与异步信号安全

信号处理函数中只能安全调用异步信号安全函数。

高频结论：

- `printf`、`malloc`、`free`、`std::cout` 通常不是异步信号安全的。
- `write` 是异步信号安全函数，handler 中输出日志常用 `write`。
- handler 中不要做复杂逻辑，通常只设置 `volatile sig_atomic_t` 标志位。

```c
volatile sig_atomic_t stopped = 0;

void on_signal(int signo) {
    stopped = 1;
}
```

### 2.4 信号屏蔽 `sigprocmask`

```c
sigset_t set;
sigemptyset(&set);
sigaddset(&set, SIGINT);
sigprocmask(SIG_BLOCK, &set, NULL);
```

重点：

- 屏蔽信号不是丢弃信号，而是延迟递送。
- 被屏蔽期间到达的普通信号通常只记录一次 pending。
- 多线程程序中更常用 `pthread_sigmask` 控制线程级信号屏蔽。

## 3. IPC 进程间通信

### 3.1 IPC 总览

| IPC | 适用场景 | 特点 |
|---|---|---|
| 匿名管道 pipe | 有亲缘关系进程 | 单向字节流，常用于父子进程 |
| 命名管道 FIFO | 无亲缘关系进程 | 文件系统中有名字，仍是字节流 |
| 消息队列 | 按消息传递 | 有消息边界，可带类型/优先级 |
| 共享内存 | 大数据高速交换 | 最快，但需要同步机制 |
| 信号量 | 同步/互斥 | 常配合共享内存使用 |
| Unix Domain Socket | 本机进程通信 | 支持双向、fd 传递、可靠通信 |

### 3.2 匿名管道 pipe

```c
int fd[2];
pipe(fd);
// fd[0] read end, fd[1] write end
```

重点：

- 管道是单向字节流。
- 常用于父子进程，因为 `fork` 后 fd 会继承。
- 读端关闭后写入可能触发 `SIGPIPE` 或返回 `EPIPE`。
- 管道容量有限，写满后阻塞，非阻塞模式下可能返回 `EAGAIN`。

### 3.3 命名管道 FIFO

```c
mkfifo("/tmp/myfifo", 0666);
```

重点：

- FIFO 在文件系统中有路径名。
- 无亲缘关系进程也能通过路径打开同一个 FIFO。
- 本质仍是管道，不是普通磁盘文件。

### 3.4 消息队列

消息队列按“消息”为单位传输数据，而不是纯字节流。

重点：

- 保留消息边界。
- POSIX 消息队列常见接口：`mq_open / mq_send / mq_receive`。
- 适合控制消息、小块结构化消息。

### 3.5 共享内存

POSIX 共享内存常见组合：

```c
int fd = shm_open("/demo", O_CREAT | O_RDWR, 0666);
ftruncate(fd, size);
void* p = mmap(NULL, size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
```

重点：

- 共享内存通常是最快 IPC，因为数据不需要在内核和用户之间反复拷贝。
- 共享内存只解决数据共享，不解决同步。
- 需要配合 mutex、semaphore、futex、文件锁等同步机制。

### 3.6 信号量 semaphore

信号量用于进程/线程间同步，表示可用资源计数。

常见操作：

- `sem_wait`：计数减 1，若为 0 则阻塞。
- `sem_post`：计数加 1，唤醒等待者。

重点：

- 二值信号量可用于互斥。
- 计数信号量适合资源池、生产者消费者模型。

### 3.7 Unix Domain Socket

Unix Domain Socket 是本机 IPC，接口类似网络 socket。

重点：

- 支持全双工通信。
- 可用于客户端/服务端模型。
- 可以传递 fd，这是普通 TCP socket 做不到的本机能力。
- 性能通常优于本机 TCP，因为不走完整网络协议栈。

## 4. `mmap`

### 4.1 `mmap` 是什么

`mmap` 把文件、设备或匿名内存映射到进程虚拟地址空间。

```c
void* addr = mmap(NULL, size, PROT_READ | PROT_WRITE,
                  MAP_SHARED, fd, 0);
```

参数速记：

| 参数 | 含义 |
|---|---|
| `addr` | 建议映射地址，通常传 `NULL` |
| `length` | 映射长度 |
| `prot` | 访问权限，如 `PROT_READ/WRITE` |
| `flags` | 共享/私有等属性 |
| `fd` | 被映射文件或设备 fd |
| `offset` | 文件偏移，通常要求页对齐 |

### 4.2 匿名映射 vs 文件映射

| 类型 | 典型 flag | 用途 |
|---|---|---|
| 匿名映射 | `MAP_ANONYMOUS` | 分配内存，不关联文件 |
| 文件映射 | 不带 `MAP_ANONYMOUS` | 把文件内容映射到内存 |

匿名映射常用于：

- 用户态大块内存分配。
- `fork` 后父子进程共享内存，常配 `MAP_SHARED | MAP_ANONYMOUS`。

文件映射常用于：

- 大文件随机访问。
- 加载共享库。
- 数据库、缓存、驱动设备内存映射。

### 4.3 `MAP_SHARED` vs `MAP_PRIVATE`

| flag | 修改是否影响底层文件/其他进程 | 常见语义 |
|---|---|---|
| `MAP_SHARED` | 可能影响 | 共享映射，适合 IPC、设备 mmap |
| `MAP_PRIVATE` | 不影响底层文件 | 私有写时复制映射 |

重点：

- `MAP_PRIVATE` 并不是一开始就复制所有内容，而是写时复制。
- `MAP_SHARED` 修改后不一定立刻落盘，落盘可用 `msync`。
- 驱动把设备 BAR 或 DMA buffer 暴露给用户态时常用 `mmap`。

### 4.4 `mmap` vs `read/write`

| 对比 | `mmap` | `read/write` |
|---|---|---|
| 访问方式 | 内存读写 | 系统调用读写 |
| 拷贝 | 可减少用户/内核拷贝 | 通常需要拷贝到用户 buffer |
| 适合 | 大文件随机访问、共享内存、设备内存 | 顺序流式读写、小文件、简单逻辑 |
| 风险 | 缺页、SIGBUS、同步复杂 | 语义直接，错误处理简单 |

高频陷阱：

- `mmap` 不总是比 `read/write` 快。
- 访问映射区可能触发缺页异常。
- 文件被截断后继续访问对应映射可能触发 `SIGBUS`。

## 5. 错误处理 `errno`

### 5.1 `errno` 基本规则

系统调用失败通常返回 `-1`，并设置 `errno` 表示错误原因。

```c
int fd = open("missing", O_RDONLY);
if (fd == -1) {
    perror("open");
}
```

重点：

- 只有在调用明确失败时才检查 `errno`。
- 成功调用不保证清零 `errno`。
- 多线程程序中，`errno` 通常是线程局部变量。

### 5.2 常见 errno

| errno | 含义 | 场景 |
|---|---|---|
| `ENOENT` | 文件或目录不存在 | `open` 不存在路径 |
| `EACCES` | 权限不足 | 无读写权限 |
| `EINTR` | 被信号中断 | 阻塞系统调用被 signal 打断 |
| `EAGAIN` | 资源暂不可用 | 非阻塞 I/O 无数据/写不进去 |
| `EBADF` | fd 非法 | 使用已关闭或无效 fd |
| `ENOMEM` | 内存不足 | 分配失败 |
| `EINVAL` | 参数非法 | flag/参数组合错误 |
| `EPIPE` | 管道破裂 | 写入无读端 pipe/socket |

### 5.3 `strerror` vs `strerror_r`

| 函数 | 特点 |
|---|---|
| `strerror(errno)` | 返回错误字符串，传统版本可能非线程安全 |
| `strerror_r` | 线程安全版本，但 GNU/POSIX 版本签名有差异 |
| `perror(msg)` | 输出 `msg: 错误原因` 到 stderr |

## 6. 高频问答

### Q1：为什么工程中更推荐 `sigaction` 而不是 `signal`？

答：`sigaction` 语义更明确，可设置信号屏蔽集和 flags，如 `SA_RESTART`，可移植性和可控性更好。

### Q2：信号处理函数里为什么不建议 `printf`？

答：`printf` 不是异步信号安全函数，内部可能使用锁和缓冲，信号打断后重入可能死锁或状态损坏。

### Q3：共享内存为什么快？

答：多个进程映射同一块物理内存，数据不需要通过内核反复拷贝；但同步要自己处理。

### Q4：`MAP_SHARED` 和 `MAP_PRIVATE` 核心区别是什么？

答：`MAP_SHARED` 的修改对其他映射者/底层对象可见；`MAP_PRIVATE` 是私有写时复制，修改不写回底层文件。

### Q5：`mmap` 一定比 `read/write` 快吗？

答：不一定。`mmap` 适合大文件随机访问和共享内存，但有缺页、同步、SIGBUS 等复杂性；顺序小文件读写 `read/write` 可能更简单甚至更快。

### Q6：什么时候检查 `errno`？

答：只有系统调用或库函数明确返回失败时才检查。成功调用不保证把 `errno` 清零。

## 9. 易错结论速背

- `SIGKILL`、`SIGSTOP` 不能捕获、阻塞或忽略。
- 普通信号可能合并，不能按次数可靠计数。
- 工程中优先用 `sigaction`，少用历史接口 `signal`。
- signal handler 中不要 `printf/malloc/free/std::cout`，复杂逻辑放主循环。
- pipe 是单向字节流：`fd[0]` 读，`fd[1]` 写。
- 共享内存最快但不负责同步，常配 semaphore/mutex。
- Unix Domain Socket 可传 fd，适合本机服务进程通信。
- `MAP_SHARED` 共享修改，`MAP_PRIVATE` 私有写时复制。
- `mmap` 适合大文件随机访问、共享内存、设备映射，但不一定总比 `read/write` 快。
- 系统调用失败看返回值，再看 `errno`；成功不保证清零 `errno`。
- `EINTR` 是被信号中断，`EAGAIN` 是暂时不可用，`EBADF` 是 fd 非法。
