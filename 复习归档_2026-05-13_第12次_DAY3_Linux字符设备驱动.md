## 2. 内核模块速记

### 2.1 最小模块结构

```c
#include <linux/init.h>
#include <linux/module.h>

static int __init hello_init(void)
{
    pr_info("hello module init\n");
    return 0;
}

static void __exit hello_exit(void)
{
    pr_info("hello module exit\n");
}

module_init(hello_init);
module_exit(hello_exit);

MODULE_LICENSE("GPL");
MODULE_AUTHOR("review");
MODULE_DESCRIPTION("minimal hello module");
```

必背：

- `module_init` 指定加载入口。
- `module_exit` 指定卸载入口。
- `MODULE_LICENSE("GPL")` 缺失时会 taint kernel，部分 GPL-only 符号不可用。
- `printk` / `pr_info` 输出看 `dmesg`。

### 2.2 模块命令

| 命令 | 作用 |
|---|---|
| `insmod hello.ko` | 加载模块，不自动解析依赖 |
| `rmmod hello` | 卸载模块 |
| `modprobe hello` | 加载模块，自动处理依赖 |
| `lsmod` | 查看已加载模块 |
| `modinfo hello.ko` | 查看模块元信息 |
| `dmesg -w` | 实时查看内核日志 |

易错：`insmod` 不处理依赖，`modprobe` 会按模块依赖加载。

## 3. 字符设备框架

### 3.1 核心对象

| 对象 | 含义 |
|---|---|
| `dev_t` | 设备号，包含 major/minor |
| major | 主设备号，定位驱动 |
| minor | 次设备号，定位同一驱动下的具体设备 |
| `struct cdev` | 内核中的字符设备对象 |
| `struct file_operations` | 用户态系统调用到驱动函数的分发表 |
| `struct class` | 用于在 `/sys/class` 下创建设备类 |
| `device_create` | 配合 udev 自动创建 `/dev/xxx` 节点 |

### 3.2 典型注册流程

```c
static dev_t devno;
static struct cdev hello_cdev;
static struct class *hello_class;

alloc_chrdev_region(&devno, 0, 1, "hello");
cdev_init(&hello_cdev, &hello_fops);
cdev_add(&hello_cdev, devno, 1);

hello_class = class_create("hello_class");
device_create(hello_class, NULL, devno, NULL, "hello");
```

卸载时反向释放：

```c
device_destroy(hello_class, devno);
class_destroy(hello_class);
cdev_del(&hello_cdev);
unregister_chrdev_region(devno, 1);
```

必背顺序：

1. 分配设备号。
2. 初始化并添加 `cdev`。
3. 创建 class。
4. 创建设备节点。
5. 卸载时逆序释放。

## 4. file_operations

### 4.1 常见回调

```c
static const struct file_operations hello_fops = {
    .owner          = THIS_MODULE,
    .open           = hello_open,
    .release        = hello_release,
    .read           = hello_read,
    .write          = hello_write,
    .unlocked_ioctl = hello_ioctl,
    .mmap           = hello_mmap,
    .poll           = hello_poll,
};
```

| 回调 | 用户态入口 | 常见用途 |
|---|---|---|
| `open` | `open("/dev/x")` | 初始化 per-file 状态、权限检查 |
| `release` | `close(fd)` | 释放 per-file 状态 |
| `read` | `read(fd, buf, len)` | 设备到用户 |
| `write` | `write(fd, buf, len)` | 用户到设备 |
| `unlocked_ioctl` | `ioctl(fd, cmd, arg)` | 控制命令、配置、提交任务 |
| `mmap` | `mmap(fd, ...)` | 映射设备内存/BAR/共享缓冲区 |
| `poll` | `poll/epoll/select` | 等待事件、可读可写、中断通知 |

### 4.2 选择题高频

- `open` 返回的是文件描述符，内核侧对应 `struct file`。
- `inode` 表示文件本身，`file` 表示一次打开实例。
- `file->private_data` 常用来挂设备私有结构或上下文。
- `.owner = THIS_MODULE` 可以防止模块使用中被卸载。
- 新内核常用 `.unlocked_ioctl`，老接口 `.ioctl` 已淘汰。

## 5. 用户/内核数据交换

### 5.1 为什么不能直接解引用用户指针

用户传入的指针属于用户虚拟地址空间，内核不能信任：

- 可能无效。
- 可能指向不可访问页。
- 可能触发缺页。
- 可能被恶意构造。

因此驱动必须用专门 API：

```c
copy_from_user(kernel_buf, user_buf, len);
copy_to_user(user_buf, kernel_buf, len);
get_user(value, user_ptr);
put_user(value, user_ptr);
```

### 5.2 返回值易错

- `copy_to_user` / `copy_from_user` 返回未拷贝成功的字节数。
- 返回 0 表示全部成功。
- 失败通常返回 `-EFAULT` 给用户态。

典型写法：

```c
if (copy_from_user(kbuf, ubuf, len))
    return -EFAULT;
```

### 5.3 上下文限制

- `copy_to_user` / `copy_from_user` 可能 sleep。
- 不能在中断上下文或持有 spinlock 的不可睡眠区域调用。
- 用户指针不要直接 `memcpy`。

## 6. 内核内存

### 6.1 kmalloc 与 vmalloc

| API | 特点 | 适合场景 |
|---|---|---|
| `kmalloc` | 虚拟连续且物理连续 | 小块内存、DMA 相关更常见 |
| `kzalloc` | `kmalloc` 后清零 | 结构体初始化 |
| `vmalloc` | 虚拟连续，物理不一定连续 | 大块内存，不要求物理连续 |
| `kmem_cache` | slab cache | 大量同类型对象 |

必背：

- `kmalloc` 物理连续，适合较小对象。
- `vmalloc` 只保证虚拟连续，物理页可以分散。
- `GFP_KERNEL` 可以睡眠，普通进程上下文使用。
- `GFP_ATOMIC` 不睡眠，中断上下文或持锁路径使用。

### 6.2 伙伴系统与 slab

- 伙伴系统管理物理页框，按 2 的幂阶分配。
- slab/slub/slob 管理小对象，减少频繁分配释放开销。
- `kmalloc` 底层通常基于 slab/slub cache。

## 7. 内核同步

### 7.1 常见同步原语

| 原语 | 是否可睡眠 | 典型场景 |
|---|---|---|
| spinlock | 不可睡眠 | 短临界区、中断上下文 |
| mutex | 可睡眠 | 进程上下文，临界区可能较长 |
| semaphore | 可睡眠 | 计数资源控制 |
| rw_semaphore | 可睡眠 | 读多写少，进程上下文 |
| atomic | 不睡眠 | 简单计数/标志 |
| RCU | 读侧轻量 | 读多写少、指针替换 |

### 7.2 高频限制

- 持有 spinlock 时不能 sleep。
- 中断上下文不能 sleep。
- mutex 不能在中断上下文使用。
- 可能 sleep 的函数不能放在 atomic context。
- RCU 读侧快，写侧需要等待 grace period。

### 7.3 判断题关键词

- “spinlock 适合长时间阻塞等待”：错。
- “mutex 可以睡眠”：对。
- “中断处理函数里可以随便调用 `copy_to_user`”：错。
- “RCU 适合读多写少”：对。

## 8. 中断处理

### 8.1 top half 与 bottom half

中断上半部：

- 由硬件中断触发。
- 要尽快执行完。
- 不能做耗时工作。
- 不能 sleep。

中断下半部：

- 延后执行耗时工作。
- 常见机制：softirq、tasklet、workqueue、threaded irq。

| 机制 | 特点 |
|---|---|
| softirq | 并发性强，网络栈常用 |
| tasklet | 基于 softirq，同一 tasklet 不并发执行，较旧 |
| workqueue | 运行在进程上下文，可以 sleep |
| threaded irq | 中断线程化，可以在 handler 线程中处理较复杂逻辑 |

### 8.2 request_irq

```c
request_irq(irq, handler, IRQF_SHARED, "devname", dev);
free_irq(irq, dev);
```

要点：

- 共享中断使用 `IRQF_SHARED`。
- handler 返回 `IRQ_HANDLED` 或 `IRQ_NONE`。
- 注册和释放要成对。
- 中断 handler 不应长时间执行。

## 9. PCIe 驱动框架

### 9.1 pci_driver

```c
static const struct pci_device_id dev_ids[] = {
    { PCI_DEVICE(VENDOR_ID, DEVICE_ID) },
    { 0, }
};
MODULE_DEVICE_TABLE(pci, dev_ids);

static struct pci_driver hello_pci_driver = {
    .name     = "hello_pci",
    .id_table = dev_ids,
    .probe    = hello_probe,
    .remove   = hello_remove,
};

module_pci_driver(hello_pci_driver);
```

### 9.2 probe 典型流程

```c
pci_enable_device(pdev);
pci_request_regions(pdev, "hello_pci");
bar0 = pci_iomap(pdev, 0, 0);

dma_set_mask_and_coherent(&pdev->dev, DMA_BIT_MASK(64));
buf = dma_alloc_coherent(&pdev->dev, size, &dma_handle, GFP_KERNEL);
```

卸载时逆序释放：

```c
dma_free_coherent(&pdev->dev, size, buf, dma_handle);
pci_iounmap(pdev, bar0);
pci_release_regions(pdev);
pci_disable_device(pdev);
```

### 9.3 BAR 与 MMIO

- BAR 是设备暴露给 CPU 的地址窗口。
- BAR 可映射寄存器或设备内存。
- 驱动通过 `pci_iomap` 映射 BAR。
- 访问 MMIO 要用 `ioread32` / `iowrite32`，不要直接普通指针解引用。
- `readl/writel` 和 `ioread32/iowrite32` 都是常见 MMIO API。

### 9.4 DMA mapping

| API | 用途 |
|---|---|
| `dma_alloc_coherent` | 分配一致性 DMA 内存 |
| `dma_map_single` | 流式 DMA 映射已有 buffer |
| `dma_unmap_single` | 解除流式 DMA 映射 |
| `dma_sync_single_for_device` | CPU 写完，交给设备前同步 |
| `dma_sync_single_for_cpu` | 设备写完，CPU 读前同步 |

易错：

- DMA 地址不是普通虚拟地址。
- non-coherent DMA 需要显式 cache 同步。
- IOMMU 可把设备 DMA 地址映射到真实物理页，提供隔离和地址转换。

## 10. mmap BAR 给用户态

### 10.1 为什么 Runtime 关心 mmap

类 CUDA Runtime 常需要高效访问设备队列、doorbell、状态页或共享缓冲区。把 BAR 或 DMA buffer 映射到用户态后，用户态 Runtime 可以减少系统调用开销。

典型路径：

1. 驱动 `probe` 发现 PCIe 设备。
2. 驱动映射 BAR，初始化队列和中断。
3. 用户态打开 `/dev/xxx`。
4. 用户态通过 `ioctl` 创建上下文/队列。
5. 用户态通过 `mmap` 映射 BAR 或共享 buffer。
6. 用户态写 command queue，敲 doorbell。
7. 设备执行任务，通过 MSI-X 中断或轮询完成通知。

### 10.2 mmap 回调常见 API

```c
remap_pfn_range(vma,
                vma->vm_start,
                phys_addr >> PAGE_SHIFT,
                size,
                vma->vm_page_prot);
```

要点：

- `mmap` 处理的是 `vm_area_struct`。
- 映射设备内存时要注意 cache 属性。
- 用户态直接访问 BAR 有安全风险，必须做权限和范围检查。
- `mmap` 不是拷贝数据，而是建立虚拟地址映射。

## 11. UIO 与 VFIO

### 11.1 UIO

UIO：Userspace I/O。

特点：

- 简化内核驱动，把大部分逻辑放到用户态。
- 内核只负责中断、mmap、基本设备暴露。
- 适合简单设备或原型开发。
- 隔离能力较弱。

### 11.2 VFIO

VFIO：Virtual Function I/O。

特点：

- 依赖 IOMMU 提供 DMA 隔离。
- 允许用户态安全访问设备资源。
- DPDK、虚拟化、用户态驱动常用。
- 设备通常绑定到 `vfio-pci`。

类 CUDA Runtime 关联：

- 用户态 Runtime 需要直接管理设备队列和内存时，VFIO 是常见路线。
- IOMMU 负责限制设备 DMA 范围，避免设备写坏任意物理内存。
- eventfd/epoll 可用于把中断事件通知给用户态。

UIO vs VFIO：

| 对比 | UIO | VFIO |
|---|---|---|
| 复杂度 | 低 | 高 |
| DMA 隔离 | 弱 | 强，依赖 IOMMU |
| 常见场景 | 简单用户态驱动 | DPDK、虚拟化、高性能用户态驱动 |
| 安全性 | 较弱 | 较强 |

## 12. 高频问答

### Q1：字符设备驱动注册的基本流程是什么？

答：分配设备号 `alloc_chrdev_region`，初始化 `cdev_init`，添加 `cdev_add`，创建 class，调用 `device_create` 生成 `/dev/xxx`，卸载时逆序释放。

### Q2：`open` 和 `release` 对应用户态什么操作？

答：`open` 对应用户态 `open()`，`release` 对应最后一次 `close()` 释放文件对象。

### Q3：为什么不能直接访问用户态指针？

答：用户指针可能无效、不可访问或恶意构造，内核必须用 `copy_to_user` / `copy_from_user` 等 API 并处理失败。

### Q4：`copy_to_user` 返回值是什么？

答：返回未成功拷贝的字节数，0 表示全部成功。

### Q5：`kmalloc` 和 `vmalloc` 区别？

答：`kmalloc` 保证物理连续，适合小块内存；`vmalloc` 保证虚拟连续，物理不一定连续，适合大块内存。

### Q6：spinlock 和 mutex 最大区别？

答：spinlock 忙等且不可睡眠，适合短临界区和中断上下文；mutex 可睡眠，只能用于允许睡眠的上下文。

### Q7：中断上半部为什么要短？

答：上半部运行在中断上下文，不能 sleep，会影响系统响应，应只做确认中断、读取状态、调度下半部等最少工作。

### Q8：PCIe BAR 是什么？

答：BAR 是 PCIe 设备暴露给主机的地址窗口，可映射设备寄存器或设备内存，驱动通过 MMIO 访问。

### Q9：为什么访问 MMIO 不能直接普通指针解引用？

答：MMIO 有特殊内存语义、顺序和屏障要求，应使用 `ioread32` / `iowrite32` 或 `readl` / `writel`。

### Q10：VFIO 为什么依赖 IOMMU？

答：用户态直接驱动设备时，设备可发起 DMA。IOMMU 用于 DMA 地址转换和隔离，防止设备访问未授权内存。

### 题 9

A. IOMMU  
B. FIFO

### 题 10

B. `.release`

答案：A

解析：驱动通过 `mmap` 建立用户虚拟地址到设备内存或共享内存的映射。

15. `dma_alloc_coherent` 分配的一致性 DMA 内存通常不需要手动 cache sync。答案：对

17. `mmap` 本质是建立映射，不是把所有数据复制一份。答案：对
18. UIO 和 VFIO 都完全不需要内核参与。答案：错
19. VFIO 常依赖 IOMMU 提供安全隔离。答案：对
20. 类 CUDA Runtime 中 doorbell 常用于通知设备有新任务。答案：对

## 15. 易错结论速背

- `insmod` 不自动处理依赖，`modprobe` 会处理依赖。
- 字符设备核心对象是 `struct cdev`。
- `file_operations` 是系统调用到驱动函数的分发表。
- `inode` 表示文件本身，`file` 表示一次打开实例。
- `file->private_data` 常挂设备私有上下文。
- 用户指针不能直接访问，要用 `copy_to_user` / `copy_from_user`。
- `copy_to_user` / `copy_from_user` 返回未拷贝成功的字节数。
- `kmalloc` 物理连续，`vmalloc` 只保证虚拟连续。
- `GFP_KERNEL` 可睡眠，`GFP_ATOMIC` 不睡眠。
- spinlock 不可睡眠，mutex 可睡眠。
- 中断上半部要短，耗时工作放下半部。
- workqueue 可以 sleep，softirq/tasklet 不能随便 sleep。
- PCIe BAR 是设备暴露给 CPU 的地址窗口。
- MMIO 用 `ioread32` / `iowrite32`，不要普通指针直接访问。
- coherent DMA 与 streaming DMA 的 cache 处理方式不同。
- `mmap` 建立映射，不是数据拷贝。
- VFIO + IOMMU 是用户态高性能驱动的重要组合。
- 类 CUDA Runtime 常见路径：command queue、doorbell、fence、MSI-X、eventfd/epoll。
