## 2. pthread 基础

### 2.1 线程 vs 进程

线程是进程内的执行流。多个线程共享同一进程的大部分资源。

| 对比 | 进程 | 线程 |
|---|---|---|
| 地址空间 | 通常独立 | 同进程线程共享 |
| 创建/切换成本 | 较高 | 较低 |
| 崩溃影响 | 通常影响本进程 | 一个线程崩溃会导致整个进程异常 |
| 通信方式 | IPC | 共享内存 + 同步 |

线程共享：

- 进程地址空间
- 全局变量、堆
- 打开的文件描述符
- 当前工作目录
- 信号处理方式

线程私有：

- 栈
- 寄存器上下文
- 线程 ID
- `errno`
- 线程局部存储 TLS

### 2.2 `pthread_create`

```c
#include <pthread.h>

void* worker(void* arg) {
    return NULL;
}

pthread_t tid;
int ret = pthread_create(&tid, NULL, worker, arg);
```

重点：

- 成功返回 0，失败返回错误码，不一定设置 `errno`。
- 线程入口函数签名是 `void* (*)(void*)`。
- 第 2 个参数是线程属性，传 `NULL` 表示默认属性。

### 2.3 `pthread_join` vs `pthread_detach`

| 操作 | 含义 | 常考点 |
|---|---|---|
| `pthread_join` | 等待线程结束并回收资源 | joinable 线程必须被 join 或 detach |
| `pthread_detach` | 分离线程，结束后自动回收 | detach 后不能再 join |

```c
pthread_join(tid, NULL);
pthread_detach(tid);
```

重点：

- 默认线程通常是 joinable。
- joinable 线程结束后，如果不 join，也不 detach，会残留线程资源。
- 一个线程不能被多个线程同时 join。

### 2.4 线程属性

`pthread_attr_t` 可设置：

- 栈大小：`pthread_attr_setstacksize`
- 分离状态：`PTHREAD_CREATE_JOINABLE / PTHREAD_CREATE_DETACHED`
- 调度策略和优先级：需要权限和系统支持

选择题常见：线程栈大小不是无限的，递归过深也可能栈溢出。

### 2.5 TLS 线程局部存储

C/C++ 常见 TLS：

```cpp
thread_local int counter = 0;
```

pthread TLS：

```c
pthread_key_t key;
pthread_key_create(&key, destructor);
pthread_setspecific(key, ptr);
void* ptr = pthread_getspecific(key);
```

重点：

- 每个线程有自己独立的一份 TLS 变量。
- `errno` 通常就是线程局部的。
- TLS 适合保存线程上下文、日志上下文、线程私有缓存。

## 3. 同步原语

### 3.1 mutex 互斥锁

mutex 用来保护临界区，保证同一时刻只有一个线程进入。

```c
pthread_mutex_lock(&mutex);
// critical section
pthread_mutex_unlock(&mutex);
```

重点：

- 锁保护的是共享数据访问，不是代码本身。
- 加锁后必须解锁，否则可能死锁。
- C++ 中优先用 RAII 锁对象管理解锁。

### 3.2 rwlock 读写锁

读写锁允许：

- 多个读者同时持有读锁。
- 写者独占持有写锁。

适用场景：读多写少。

```c
pthread_rwlock_rdlock(&rwlock);
pthread_rwlock_wrlock(&rwlock);
pthread_rwlock_unlock(&rwlock);
```

注意：

- 写多场景下读写锁不一定比 mutex 更快。
- 实现策略不同，可能有读者饥饿或写者饥饿问题。

### 3.3 spinlock 自旋锁

自旋锁获取不到锁时不会睡眠，而是在 CPU 上忙等。

适用场景：

- 临界区极短。
- 持锁线程很快释放。
- 不希望线程睡眠/唤醒带来上下文切换成本。

不适合：

- 临界区长。
- 持锁期间可能阻塞。
- 单核 CPU 上长时间自旋。

选择题记忆：自旋锁不是“更高级的 mutex”，用错会浪费 CPU。

### 3.4 条件变量 condition variable

条件变量用于线程等待某个条件成立，必须配合 mutex 使用。

```c
pthread_mutex_lock(&mutex);
while (!ready) {
    pthread_cond_wait(&cond, &mutex);
}
// ready == true
pthread_mutex_unlock(&mutex);
```

重点：

- `pthread_cond_wait` 会原子地释放 mutex 并睡眠。
- 被唤醒后会重新获取 mutex 再返回。
- 必须用 `while` 检查条件，不能用 `if`。
- 原因：虚假唤醒、多个等待者竞争、条件可能被其他线程改变。

`signal` vs `broadcast`：

| 操作 | 含义 |
|---|---|
| `pthread_cond_signal` | 唤醒至少一个等待线程 |
| `pthread_cond_broadcast` | 唤醒所有等待线程 |

### 3.5 semaphore 信号量

信号量维护一个计数。

```c
sem_wait(&sem); // P 操作，计数减 1，不够则阻塞
sem_post(&sem); // V 操作，计数加 1，唤醒等待者
```

用途：

- 限制并发资源数量。
- 生产者消费者计数。
- 进程间同步。

区别：

- mutex 表示互斥所有权，谁 lock 通常谁 unlock。
- semaphore 更像资源计数，`sem_post` 不一定由 `sem_wait` 的同一线程执行。

## 4. C++11 并发基础

### 4.1 `std::thread`

```cpp
#include <thread>

std::thread t([] {
    // work
});

t.join();
```

重点：

- `std::thread` 构造后立即启动线程。
- `join()` 等待线程结束。
- `detach()` 分离线程，后台执行。
- `std::thread` 对象析构时如果仍 joinable，会调用 `std::terminate()`。

高频陷阱：

```cpp
std::thread t(work);
// 如果既不 join 也不 detach，t 析构会 terminate
```

### 4.2 `std::mutex` 与 RAII

```cpp
std::mutex mutex;

void f() {
    std::lock_guard<std::mutex> lock(mutex);
    // critical section
}
```

`lock_guard`：

- 构造时加锁。
- 析构时解锁。
- 简单、不可手动 unlock。

`unique_lock`：

- 更灵活，可延迟加锁、提前解锁、转移所有权。
- 条件变量 `std::condition_variable` 必须配合 `std::unique_lock<std::mutex>`。

### 4.3 `std::condition_variable`

```cpp
std::mutex mutex;
std::condition_variable cv;
bool ready = false;

std::unique_lock<std::mutex> lock(mutex);
cv.wait(lock, [] { return ready; });
```

重点：

- `wait(lock, pred)` 等价于循环检查条件，能抵抗虚假唤醒。
- 通知方一般在修改共享状态后 `notify_one` 或 `notify_all`。
- 条件变量不是事件本身，真正条件必须由共享变量表达。

### 4.4 `future / promise / async`

`promise` 负责设置结果，`future` 负责获取结果。

```cpp
std::promise<int> p;
std::future<int> f = p.get_future();
p.set_value(42);
int x = f.get();
```

`std::async`：

```cpp
auto f = std::async(std::launch::async, [] {
    return 42;
});
int result = f.get();
```

重点：

- `future::get()` 只能调用一次。
- `get()` 会阻塞直到结果就绪，并重新抛出异步任务中的异常。
- `std::async` 默认策略可能异步执行，也可能延迟执行；想强制新线程可用 `std::launch::async`。

### 4.5 `std::atomic` 基础

```cpp
std::atomic<int> counter{0};
counter.fetch_add(1);
```

重点：

- atomic 操作避免数据竞争。
- `counter++` 对普通 int 不是原子操作。
- atomic 只保证该变量自身操作的原子性，不自动保护多个变量之间的不变量。
- 默认内存序是 `memory_order_seq_cst`，最强也最容易理解。

## 5. 高频问答

### Q1：`pthread_create` 成功和失败分别返回什么？

答：成功返回 0，失败返回错误码；pthread 系列函数很多不是通过 `errno` 返回错误。

### Q2：`join` 和 `detach` 的区别？

答：`join` 等待线程结束并回收资源；`detach` 分离线程，线程结束后系统自动回收，detach 后不能再 join。

### Q3：什么时候用读写锁？

答：读多写少时可能适合读写锁，因为多个读者可并发；写多或临界区很短时不一定比 mutex 好。

### Q4：什么时候用自旋锁？

答：临界区极短、预计很快释放、上下文切换成本比忙等更高时。临界区长或可能阻塞时不要用。

### Q5：条件变量为什么要用 `while` 而不是 `if`？

答：因为存在虚假唤醒、多个线程被唤醒后竞争、条件被其他线程抢先改变等情况，醒来必须重新检查条件。

### Q6：`std::thread` 析构时还 joinable 会怎样？

答：会调用 `std::terminate()`，所以必须在析构前 `join()` 或 `detach()`。

### Q7：`lock_guard` 和 `unique_lock` 的区别？

答：`lock_guard` 简单 RAII 锁，构造加锁析构解锁；`unique_lock` 更灵活，支持延迟锁、提前解锁、移动，并可用于条件变量。

### Q8：atomic 能替代 mutex 吗？

答：不能简单替代。atomic 适合单变量或简单无锁状态，复杂不变量、多变量一致性通常仍需要 mutex。

## 6. 选择题训练

### 题 1

`pthread_create` 成功时返回：

A. `0`  
B. `1`  
C. 新线程返回值  
D. `errno`

答案：A

解析：pthread 系列函数成功通常返回 0，失败返回错误码。

### 题 2

默认情况下 pthread 线程通常是：

A. detached  
B. joinable  
C. 僵尸进程  
D. 内核模块

答案：B

解析：默认线程通常是 joinable，需要 join 或 detach。

### 题 3

`pthread_detach` 后的线程：

A. 仍然可以被 `pthread_join`  
B. 结束后资源自动回收  
C. 会立刻停止执行  
D. 变成进程

答案：B

解析：detached 线程结束后自动回收，不能再 join。

### 题 4

读写锁最适合：

A. 写多读少  
B. 读多写少  
C. 单线程程序  
D. 永远不释放锁

答案：B

解析：读写锁允许多个读者并发，适合读多写少。

### 题 5

自旋锁获取不到锁时通常会：

A. 立即退出进程  
B. 在 CPU 上忙等  
C. 自动释放所有锁  
D. 创建子进程

答案：B

解析：spinlock 的特点是忙等，不睡眠。

### 题 6

条件变量等待条件时，推荐使用：

A. `if` 检查一次  
B. `while` 循环检查  
C. 不检查条件  
D. 只检查线程 ID

答案：B

解析：条件变量可能虚假唤醒，醒来后必须重新检查条件。

### 题 7

`std::thread` 对象析构时如果仍 joinable，会：

A. 自动 join  
B. 自动 detach  
C. 调用 `std::terminate()`  
D. 什么也不做

答案：C

解析：这是 C++11 线程最常考陷阱之一。

### 题 8

`std::condition_variable` 通常需要配合：

A. `std::unique_lock<std::mutex>`  
B. `std::string`  
C. `std::vector<int>`  
D. `std::cout`

答案：A

解析：`condition_variable::wait` 要求 `unique_lock<mutex>`。

### 题 9

`std::future::get()` 的特点是：

A. 可无限次调用  
B. 不会阻塞  
C. 通常只能调用一次，可能阻塞等待结果  
D. 会创建进程

答案：C

解析：`future::get()` 获取结果，通常只能调用一次。

### 题 10

普通 `int counter++` 在多线程下：

A. 一定原子  
B. 不是原子操作，可能数据竞争  
C. 等价于 `std::atomic<int>`  
D. 会自动加锁

答案：B

解析：普通自增包含读、改、写，不是原子操作。

## 8. 易错结论速背

- 线程共享地址空间、堆、全局变量、fd；私有栈、寄存器、TLS。
- `pthread_create` 成功返回 0，失败返回错误码。
- joinable 线程要么 join，要么 detach。
- detach 后不能 join；join 后也不能重复 join。
- 读写锁适合读多写少，不是万能加速。
- 自旋锁忙等，适合极短临界区，用错会烧 CPU。
- 条件变量必须配 mutex，等待条件必须用 `while`。
- `pthread_cond_wait` 会原子释放 mutex 并睡眠，醒来后重新加锁。
- `std::thread` 析构时仍 joinable 会 `std::terminate()`。
- `lock_guard` 简单，`unique_lock` 灵活，条件变量用 `unique_lock`。
- `future::get()` 通常只能调用一次。
- `std::async` 默认不保证一定新线程，强制异步用 `std::launch::async`。
- atomic 适合简单原子操作，不等价于自动保护所有共享状态。
