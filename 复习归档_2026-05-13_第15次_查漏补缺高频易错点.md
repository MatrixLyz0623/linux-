## 1. C/C++ 零散高频

### 1.1 `struct` 与 `class`

| 场景 | 区别 |
|---|---|
| C++ `struct` vs `class` | 默认访问权限不同：`struct` 默认 public，`class` 默认 private |
| C++ 继承默认权限 | `struct` 默认 public 继承，`class` 默认 private 继承 |
| C `struct` | 只有数据聚合，无成员函数、访问控制、继承、多态 |

判断题：C++ 中 `struct` 不能有成员函数。答案：错。

### 1.2 `static` 的多重含义

| 位置 | 含义 |
|---|---|
| 函数内局部变量 | 静态存储期，只初始化一次，生命周期到程序结束 |
| 全局变量/函数前 | internal linkage，仅当前翻译单元可见 |
| 类静态成员变量 | 属于类，不属于某个对象，类外通常要定义 |
| 类静态成员函数 | 无 `this` 指针，只能直接访问静态成员 |

易错：函数内 `static` 局部变量在 C++11 起初始化是线程安全的。

### 1.3 `const` / `constexpr` / `volatile`

- `const` 表示不可通过该名字修改，不等于一定在编译期可知。
- `constexpr` 要求可用于编译期求值的语境。
- `volatile` 表示对象可能被编译器不可见的方式修改，禁止某些优化。
- `volatile` 不是线程同步原语，不能替代 `atomic`。

判断题：`volatile int` 可以保证多线程自增安全。答案：错。

### 1.4 `inline` 真正含义

- 现代 C++ 中 `inline` 更重要的语义是允许多翻译单元中出现相同定义，避免 ODR 链接冲突。
- 编译器是否内联展开由优化器决定，不是写了 `inline` 就一定展开。
- 类内定义的成员函数隐式 inline。

### 1.5 `extern "C"`

作用：关闭 C++ name mangling，让 C++ 能按 C ABI 暴露/引用符号。

常见写法：

```cpp
#ifdef __cplusplus
extern "C" {
#endif

void c_api_func(int x);

#ifdef __cplusplus
}
#endif
```

易错：`extern "C"` 不会把 C++ 代码变成 C 代码，只影响链接符号名和调用约定相关 ABI。

### 1.6 `shared_ptr` 线程安全边界

- 控制块引用计数增减是线程安全的。
- 多个 `shared_ptr` 实例同时管理同一个对象，引用计数不会乱。
- 被管理对象本身不是自动线程安全。
- 同一个 `shared_ptr` 对象被多个线程同时读写，也需要同步，或使用 `std::atomic<std::shared_ptr<T>>`。

判断题：`shared_ptr` 指向的对象天然线程安全。答案：错。

## 2. 编译、链接、库与 ELF

### 2.1 编译流程

```text
.c/.cpp -> 预处理 -> 编译 -> 汇编 -> 链接 -> 可执行文件
```

对应选项：

| 选项 | 阶段 |
|---|---|
| `-E` | 只预处理 |
| `-S` | 生成汇编 `.s` |
| `-c` | 生成目标文件 `.o`，不链接 |
| `-o` | 指定输出文件 |

### 2.2 静态库 vs 动态库

| 类型 | 后缀 | 特点 |
|---|---|---|
| 静态库 | `.a` | 链接时拷贝需要的目标代码进可执行文件，运行时不依赖 `.a` |
| 动态库 | `.so` | 运行时加载，多个进程可共享代码段，需要能被动态链接器找到 |

易错：静态库 `.a` 运行时不需要继续存在；动态库 `.so` 运行时需要能找到。

### 2.3 链接顺序

GNU ld 常按从左到右解析符号。

```bash
g++ main.o -lfoo   # 常见正确：main.o 依赖 libfoo
g++ -lfoo main.o   # 可能失败：扫描 libfoo 时还没看到未解析符号
```

结论：被依赖的库放后面。

### 2.4 `-pthread` vs `-lpthread`

- `-pthread` 是编译和链接选项，会定义线程相关宏并链接 pthread。
- `-lpthread` 只是链接库。
- 实战优先用 `-pthread`。

### 2.5 `-fPIC` 与动态库

- `-fPIC` 生成位置无关代码。
- 构建 `.so` 通常需要 `-fPIC`。
- 位置无关代码便于动态库被映射到不同虚拟地址。

### 2.6 ELF 常见段

| 段 | 内容 |
|---|---|
| `.text` | 代码 |
| `.rodata` | 只读常量 |
| `.data` | 已初始化全局/静态变量 |
| `.bss` | 未初始化或零初始化全局/静态变量 |
| `.plt` / `.got` | 动态链接跳转和符号解析相关 |

常用命令：

```bash
file ./app
ldd ./app
readelf -a ./app
objdump -d ./app
nm -D libfoo.so
strings ./app
```

## 3. 程序从 main 到 exit

真正入口通常不是 `main`，而是 `_start`。

简化链路：

```text
内核加载 ELF
  -> 动态链接器装载依赖库
  -> _start
  -> C/C++ runtime 初始化
  -> 全局/静态对象构造
  -> main(argc, argv, envp)
  -> return 或 exit
  -> atexit 回调
  -> 全局/静态对象析构
  -> 进程退出，内核回收资源
```

易错：

- `main` 不是进程第一条用户态指令。
- `_exit` 直接系统调用退出，不刷新 stdio 缓冲，不跑 `atexit`。
- `exit` 会做 C 库清理，刷新缓冲，调用 `atexit`。

## 4. Linux 文件系统与权限陷阱

### 4.1 inode / dentry / 文件名

- inode 存元数据和数据块索引。
- 文件名存放在父目录的目录项 dentry 中。
- ext4 inode 不直接存文件名。

判断题：inode 中保存文件名。答案：错。

### 4.2 硬链接与软链接

| 类型 | 特点 |
|---|---|
| 硬链接 | 多个目录项指向同一 inode，不能跨文件系统，普通用户不能给目录建硬链接 |
| 软链接 | 独立 inode，内容是路径字符串，可跨文件系统，可指向目录 |

删除文件：本质是减少目录项链接。若链接数为 0 且没有进程打开 fd，空间才释放。

### 4.3 文件删除但空间不释放

原因：文件已 unlink，但仍有进程持有 fd。

排查：

```bash
lsof | grep deleted
```

处理：重启/关闭持有 fd 的进程，或让进程重新打开日志文件。

### 4.4 目录权限

- 目录 `r`：能列出目录项名称。
- 目录 `w`：能创建/删除/重命名目录项，通常还需要 `x`。
- 目录 `x`：能进入目录、访问目录下已知名称的文件。

判断题：目录没有 x 权限也可以 `cd` 进去。答案：错。

### 4.5 特殊权限位

| 位 | 数字 | 作用 |
|---|---:|---|
| setuid | 4xxx | 执行时以文件所有者身份运行 |
| setgid | 2xxx | 执行时以文件所属组身份运行；目录中新文件继承组 |
| sticky bit | 1xxx | 目录中只有文件所有者/root 可删除自己的文件，如 `/tmp` |

### 4.6 umask

- 新文件默认基准权限 666。
- 新目录默认基准权限 777。
- `umask 022`：新文件 644，新目录 755。

## 5. Shell 与命令行易错点

### 5.1 重定向顺序

```bash
cmd > out 2>&1   # stdout 到 out，然后 stderr 复制 stdout，也到 out
cmd 2>&1 > out   # stderr 先复制当前 stdout，仍在终端；stdout 后到 out
```

结论：重定向从左到右生效。

### 5.2 fd 编号

- 0：stdin。
- 1：stdout。
- 2：stderr。

### 5.3 Shell 变量与判断

```bash
var=value      # 对
var = value    # 错，等号两边不能有空格
```

`[` 是命令，前后必须有空格：

```bash
if [ "$x" -eq 0 ]; then
    echo ok
fi
```

### 5.4 `set -euo pipefail`

| 选项 | 含义 |
|---|---|
| `set -e` | 命令失败时退出 |
| `set -u` | 使用未定义变量时报错 |
| `set -o pipefail` | 管道中任一命令失败则整体失败 |

### 5.5 子 shell 与当前 shell

```bash
(cd /tmp; pwd)   # 子 shell，不影响当前目录
{ cd /tmp; pwd; } # 当前 shell，会影响当前目录，注意空格和分号
```

判断题：`()` 和 `{}` 都一定在当前 shell 执行。答案：错。

## 6. 进程、信号与系统管理补点

### 6.1 kill 不一定能杀死进程

`kill -9` 发送 SIGKILL，不能被捕获或忽略，但仍有杀不掉的场景：

- 僵尸进程：已经退出，只剩进程表项，等父进程回收。
- D 状态：不可中断睡眠，通常在等待内核 I/O。

### 6.2 僵尸进程与孤儿进程

- 僵尸进程：子进程退出，父进程没有 `wait`，保留退出状态。
- 孤儿进程：父进程先退出，子进程被 init/systemd 接管。

### 6.3 `fork` 后文件描述符

`fork` 后子进程获得父进程 fd 表副本，但 fd 指向同一个打开文件表项，因此共享文件偏移。

判断题：`fork` 后父子进程同一个 fd 对应的文件偏移完全独立。答案：错。

### 6.4 `ps` 看线程

```bash
ps -eLf
top -H -p <pid>
```

Linux 线程本质是共享资源更多的 task，也称 LWP。

## 7. Git / Make / Vim 必背坑

### 7.1 Git

| 命令 | 作用 |
|---|---|
| `git reset --soft HEAD~1` | 撤销提交，改动留在暂存区 |
| `git reset --mixed HEAD~1` | 撤销提交，改动留在工作区 |
| `git reset --hard HEAD~1` | 撤销提交并丢弃改动，危险 |
| `git revert <commit>` | 新增一个反向提交，安全撤销公开历史 |
| `git fetch` | 拉远端信息，不合并 |
| `git pull` | fetch + merge/rebase |

结论：已 push 的公共提交不要随便 rebase 或 reset --hard。

### 7.2 Makefile

- 命令行必须以 Tab 开头。
- `$@`：目标。
- `$<`：第一个依赖。
- `$^`：所有依赖。
- `:=` 立即展开。
- `=` 延迟展开。
- `.PHONY` 防止伪目标和同名文件冲突。

### 7.3 Vim

- `i/a/o` 进插入。
- `Esc` 回普通模式。
- `dd` 删除行。
- `yy` 复制行。
- `p` 粘贴。
- `u` 撤销。
- `Ctrl-r` 重做。
- `:%s/old/new/g` 全局替换。
- `:wq` 保存退出，`:q!` 强退。

## 8. 网络补点

### 8.1 大端小端

- 大端：高位字节存低地址。
- 小端：低位字节存低地址。
- 网络字节序是大端。

判断函数：

```c
int is_little_endian(void)
{
    unsigned int x = 1;
    return *(unsigned char *)&x == 1;
}
```

### 8.2 `SO_REUSEADDR` vs `SO_REUSEPORT`

| 选项 | 重点 |
|---|---|
| `SO_REUSEADDR` | 允许地址复用，常用于服务重启时绑定 TIME_WAIT 端口等场景 |
| `SO_REUSEPORT` | 多个 socket 绑定同一 IP:port，内核做负载分发 |

二者不是同义词。

### 8.3 `sendfile` / `splice` / `mmap`

- `mmap`：把文件映射进进程虚拟地址空间，减少 read/write 拷贝路径。
- `sendfile`：文件到 socket 的内核态转发，减少用户态拷贝。
- `splice`：在 pipe 和 fd 之间移动数据，支持零拷贝管线。

选择题：静态文件服务器常用 `sendfile` 减少拷贝。

## 9. 异构 / GPU 追问补点

### 9.1 Occupancy

Occupancy 通常指 SM 上活跃 warp 数相对最大可容纳 warp 数的比例。

影响因素：

- 每线程寄存器数量。
- 每 block shared memory 用量。
- block size。
- 硬件最大线程/warp/block 限制。

易错：Occupancy 越高不一定性能越高，只是隐藏延迟的一个条件。

### 9.2 Roofline 模型

核心：性能受计算峰值和内存带宽共同限制。

```text
Arithmetic Intensity = FLOPs / Byte
```

- 算术强度低：memory-bound。
- 算术强度高：compute-bound。

### 9.3 Kernel launch overhead

小 kernel 很多时，启动开销可能成为瓶颈。

优化：

- kernel fusion。
- graph capture / CUDA Graph 类机制。
- persistent kernel。
- batch submit。

### 9.4 NCCL / Collective

AI 训练和多卡推理常见通信：

- AllReduce。
- AllGather。
- ReduceScatter。
- Broadcast。

常见拓扑：ring、tree、hierarchical。

自研 Runtime 若要接大模型生态，通信库是重要一环。

## 12. 最后一页速背

- `struct` 默认 public，`class` 默认 private。
- `volatile` 不是线程同步，不替代 `atomic`。
- `inline` 不保证内联展开，更重要是 ODR 相关语义。
- `extern "C"` 防止 C++ name mangling。
- `shared_ptr` 控制块安全，对象本身不安全。
- `-c` 只编译不链接，`-fPIC` 常用于动态库。
- 静态库运行时不需要，动态库运行时要能找到。
- 被依赖库放链接命令后面。
- `_start` 早于 `main`，`exit` 不等于 `_exit`。
- 文件名在目录项，inode 不直接存文件名。
- 删除文件但 fd 仍打开，空间不释放。
- 目录 x 权限表示可进入。

- `fork` 后 fd 指向同一打开文件表项，共享偏移。

- Makefile 命令行必须 Tab。
- 网络字节序大端。
- `SO_REUSEPORT` 用于多 socket 同端口负载分发。
- `sendfile` 是静态文件服务器零拷贝高频点。
- GPU occupancy 不是越高越好，Roofline 看算术强度。
