## 2. TCP 三次握手

### 2.1 流程

```text
Client                                Server
  | -------- SYN, seq=x ------------> |
  | <---- SYN+ACK, seq=y, ack=x+1 --- |
  | -------- ACK, ack=y+1 ----------> |
```

三次握手目的：

- 确认双方收发能力。
- 协商初始序列号 ISN。
- 建立连接状态。

### 2.2 为什么不是两次握手

两次握手无法让服务端确认客户端已经收到自己的 SYN+ACK，也容易让旧的重复 SYN 报文导致服务端误建立连接。

### 2.3 状态变化

客户端：

```text
CLOSED -> SYN_SENT -> ESTABLISHED
```

服务端：

```text
CLOSED -> LISTEN -> SYN_RCVD -> ESTABLISHED
```

易错：服务端收到 SYN 后进入 `SYN_RCVD`，不是直接 `ESTABLISHED`。

## 3. TCP 四次挥手

### 3.1 流程

```text
Active Close                          Passive Close
  | -------- FIN -------------------> |
  | <------- ACK -------------------- |
  | <------- FIN -------------------- |
  | -------- ACK -------------------> |
```

四次原因：TCP 是全双工。主动方关闭发送方向后，被动方可能还有数据要发送，所以 ACK 和 FIN 通常分开发。

### 3.2 状态变化

主动关闭方：

```text
ESTABLISHED -> FIN_WAIT_1 -> FIN_WAIT_2 -> TIME_WAIT -> CLOSED
```

被动关闭方：

```text
ESTABLISHED -> CLOSE_WAIT -> LAST_ACK -> CLOSED
```

### 3.3 TIME_WAIT

TIME_WAIT 出现在主动关闭方。

作用：

- 保证最后一个 ACK 能被对端收到。
- 让旧连接的残留报文在网络中消失，避免污染新连接。

持续时间：通常是 2MSL。

易错：

- TIME_WAIT 多不一定是 bug，常见于主动关闭连接的一方。
- 服务端如果主动关闭短连接，也可能出现大量 TIME_WAIT。

### 3.4 CLOSE_WAIT

CLOSE_WAIT 出现在被动关闭方，表示：对端已经 FIN，本端应用还没 close。

大量 CLOSE_WAIT 通常说明应用层没有及时关闭 fd，常见原因：

- 业务逻辑漏 `close`。
- 线程卡住。
- 连接生命周期管理错误。

判断题：大量 CLOSE_WAIT 一般是本端程序问题，不是网络自然现象。

## 4. select / poll / epoll

### 4.1 对比表

| 机制 | fd 集合 | fd 数量限制 | 拷贝开销 | 就绪查找 | 典型特点 |
|---|---|---:|---|---|---|
| `select` | `fd_set` 位图 | 通常 1024，受 `FD_SETSIZE` 影响 | 每次拷贝 | O(n) 扫描 | 老接口，限制多 |
| `poll` | `pollfd` 数组 | 无固定 1024 限制 | 每次拷贝 | O(n) 扫描 | 比 select 灵活 |
| `epoll` | 内核维护 interest set | 无 1024 限制 | 避免每次全量传入 | 返回就绪事件 | Linux 高并发常用 |

### 4.2 select 易错点

- `select` 会修改传入的 `fd_set`，循环中要重新设置。
- 最大 fd 参数是 `maxfd + 1`。
- fd 数量受 `FD_SETSIZE` 限制。
- 每次调用都要把 fd 集合从用户态拷到内核态。

### 4.3 poll 易错点

- `poll` 不受 `FD_SETSIZE` 固定限制。
- 但仍需要线性扫描 `pollfd` 数组。
- 大量 fd 下性能仍受 O(n) 影响。

### 4.4 epoll 三个 API

```c
int epoll_create1(int flags);
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);
```

`epoll_ctl` 操作：

- `EPOLL_CTL_ADD`
- `EPOLL_CTL_MOD`
- `EPOLL_CTL_DEL`

### 4.5 epoll 内部结构

常见面试说法：

- 红黑树：维护关注的 fd，也就是 interest set。
- 就绪链表：保存已经就绪的 fd。
- `epoll_wait` 主要从就绪链表取事件。

注意：说 `epoll` 完全 O(1) 是过度简化，事件触发、回调、用户拷贝仍有成本。

## 5. LT 与 ET

### 5.1 LT：Level Triggered，水平触发

只要 fd 仍处于就绪状态，`epoll_wait` 就会反复通知。

特点：

- 默认模式。
- 编程简单。
- 即使一次没读完，后续还会通知。

### 5.2 ET：Edge Triggered，边沿触发

状态从未就绪变为就绪时才通知一次。

特点：

- 通常必须配合非阻塞 fd。
- 收到事件后要循环读/写直到 `EAGAIN`。
- 如果没读干净，可能不再通知，导致连接卡住。

典型读法：

```c
while (true) {
    ssize_t n = read(fd, buf, sizeof(buf));
    if (n > 0) {
        handle(buf, n);
    } else if (n == -1 && errno == EAGAIN) {
        break;
    } else {
        close(fd);
        break;
    }
}
```

易错：ET 不是“比 LT 一定更快”，它减少重复通知，但要求代码正确 drain fd。

## 6. Reactor 与 Proactor

### 6.1 Reactor

Reactor：同步非阻塞 I/O + 事件通知。

流程：

1. `epoll_wait` 等待 fd 就绪。
2. Reactor 分发事件。
3. 业务 handler 自己执行 `read/write`。
4. 处理完成后继续等待。

常见组合：Linux `epoll` + nonblocking socket。

### 6.2 Proactor

Proactor：异步 I/O 完成通知。

流程：

1. 应用提交异步读写。
2. 内核或系统完成 I/O。
3. 通知应用完成事件。
4. 应用处理完成后的数据。

典型：Windows IOCP 更接近 Proactor。

Linux 上很多高性能网络库以 Reactor 为主。

### 6.3 对比

| 模型 | 通知含义 | 谁执行 I/O | 常见实现 |
|---|---|---|---|
| Reactor | fd 已就绪 | 应用线程 | epoll + nonblocking socket |
| Proactor | I/O 已完成 | 内核/系统异步机制 | IOCP，部分 io_uring 场景 |

选择题关键词：Reactor 是“就绪通知”，Proactor 是“完成通知”。

## 7. 惊群与 SO_REUSEPORT

### 7.1 惊群问题

惊群：多个进程/线程等待同一个事件，一个连接到来时多个等待者被唤醒，但只有一个能处理，其他被白白唤醒。

典型场景：

- 多进程 accept 同一个 listen fd。
- 多线程同时等待同一个事件源。

影响：

- 上下文切换增加。
- 锁竞争增加。
- CPU 浪费。

### 7.2 缓解方式

- 内核层面改进 wake one。
- `EPOLLEXCLUSIVE`。
- 主从 Reactor。
- 每个进程独立 listen socket + `SO_REUSEPORT`。

### 7.3 SO_REUSEPORT

`SO_REUSEPORT` 允许多个 socket 绑定相同 IP:port，内核把连接分发到不同 socket。

好处：

- 多进程负载均衡 accept。
- 减少共享 listen fd 竞争。
- 可结合 CPU 亲和性。

易错：`SO_REUSEADDR` 和 `SO_REUSEPORT` 不是一回事。

## 8. HTTP 基础

### 8.1 HTTP 特点

- 应用层协议。
- 基于请求-响应模型。
- HTTP/1.1 默认长连接。
- HTTP 是无状态协议。
- 常见方法：GET、POST、PUT、DELETE、HEAD、OPTIONS。

### 8.2 状态码

| 状态码 | 含义 |
|---|---|
| 200 | OK |
| 301 | 永久重定向 |
| 302 | 临时重定向 |
| 400 | Bad Request |
| 401 | Unauthorized |
| 403 | Forbidden |
| 404 | Not Found |
| 500 | Internal Server Error |
| 502 | Bad Gateway |
| 503 | Service Unavailable |

### 8.3 HTTP/1.0、1.1、2、3

| 版本 | 关键点 |
|---|---|
| HTTP/1.0 | 默认短连接 |
| HTTP/1.1 | 默认长连接、管线化但有队头阻塞 |
| HTTP/2 | 二进制分帧、多路复用、头部压缩，仍基于 TCP |
| HTTP/3 | 基于 QUIC/UDP，减少 TCP 队头阻塞影响 |

## 9. HTTPS 基础

HTTPS = HTTP + TLS。

提供：

- 机密性：加密。
- 完整性：防篡改。
- 身份认证：证书链。

### 9.1 TLS 握手核心思想

非对称加密用于身份认证和密钥协商，对称加密用于后续数据传输。

原因：

- 非对称加密成本高。
- 对称加密性能好。

### 9.2 证书链

浏览器验证：

- 证书是否由可信 CA 签发。
- 域名是否匹配。
- 是否过期。
- 是否被吊销。

易错：HTTPS 不保证服务器业务没有 bug，只保证传输安全和身份认证。

## 10. 高频问答

### Q1：为什么 TCP 建立连接是三次握手？

答：三次握手能确认双方收发能力，并协商初始序列号。两次握手无法确认客户端收到服务端的 SYN+ACK，旧 SYN 也可能导致错误连接。

### Q2：为什么挥手通常是四次？

答：TCP 是全双工，一方关闭发送方向后，另一方可能仍有数据要发，所以 ACK 和 FIN 通常分开发。

### Q3：TIME_WAIT 在哪一方？有什么作用？

答：TIME_WAIT 在主动关闭方，用于保证最后 ACK 可重传，并让旧报文在网络中消失。

### Q4：大量 CLOSE_WAIT 说明什么？

答：通常说明本端应用收到对端 FIN 后没有及时 close fd，是应用层资源管理问题。

### Q5：select、poll、epoll 的核心区别？

答：select 有 fd_set 限制并线性扫描；poll 用数组无固定 1024 限制但仍线性扫描；epoll 在内核维护关注集合和就绪链表，更适合大量连接。

### Q6：epoll 的 LT 和 ET 区别？

答：LT 只要就绪就持续通知；ET 只在状态变化时通知，要求非阻塞并读写到 `EAGAIN`。

### Q7：Reactor 和 Proactor 区别？

答：Reactor 通知“可以 I/O 了”，应用自己读写；Proactor 通知“I/O 已完成”，应用处理结果。

### Q8：惊群是什么？

答：多个等待者被同一事件唤醒，但只有一个能处理，造成无效唤醒、竞争和上下文切换。

### Q9：HTTP 和 HTTPS 区别？

答：HTTPS 在 HTTP 下增加 TLS，提供加密、完整性和身份认证。

### Q10：HTTP/2 和 HTTP/3 的关键区别？

答：HTTP/2 基于 TCP，支持二进制分帧和多路复用；HTTP/3 基于 QUIC/UDP，改善 TCP 队头阻塞问题。

## 11. 选择题训练

### 题 1

TCP 三次握手中，服务端收到 SYN 后通常进入：

A. ESTABLISHED  
B. SYN_RCVD  
C. TIME_WAIT  
D. CLOSE_WAIT

答案：B

解析：服务端收到 SYN 并发送 SYN+ACK 后进入 SYN_RCVD。

### 题 2

TIME_WAIT 通常出现在：

A. 主动关闭方  
B. 被动关闭方  
C. 只在服务端  
D. 只在 UDP

答案：A

解析：TIME_WAIT 是主动关闭方等待 2MSL。

### 题 3

大量 CLOSE_WAIT 通常说明：

A. 本端应用没有及时关闭连接  
B. 对端一定宕机  
C. DNS 解析失败  
D. UDP 丢包

答案：A

解析：CLOSE_WAIT 是被动关闭方等待应用 close。

### 题 4

`select` 的一个经典限制是：

A. 只能处理 UDP  
B. fd 数量受 `FD_SETSIZE` 影响  
C. 只能用于磁盘文件  
D. 不能设置超时

答案：B

解析：`select` 使用 `fd_set`，常见限制为 1024。

### 题 5

`epoll` 中维护关注 fd 集合的结构常被描述为：

A. 红黑树  
B. 哈希文件系统  
C. B+ 树索引  
D. 栈

答案：A

解析：面试中常说 epoll 用红黑树维护 interest set。

### 题 6

ET 模式下正确做法通常是：

A. 阻塞读一次就结束  
B. 使用非阻塞 fd 并读到 `EAGAIN`  
C. 不需要处理错误  
D. 关闭所有事件

答案：B

解析：ET 只通知状态变化，不 drain 可能卡住。

### 题 7

Reactor 模型通知应用的是：

A. I/O 已完成  
B. fd 已就绪  
C. 证书已过期  
D. 线程已销毁

答案：B

解析：Reactor 是就绪通知。

### 题 8

`SO_REUSEPORT` 的常见作用是：

A. 多 socket 绑定同一端口并由内核分发连接  
B. 加密 HTTP 内容  
C. 修改 TCP 序列号  
D. 禁用 epoll

答案：A

解析：`SO_REUSEPORT` 常用于多进程负载均衡 accept。

### 题 9

HTTPS 相比 HTTP 增加的是：

A. TLS  
B. FTP  
C. DNS  
D. ICMP

答案：A

解析：HTTPS = HTTP over TLS。

### 题 10

HTTP/3 基于：

A. TCP  
B. QUIC/UDP  
C. ICMP  
D. ARP

答案：B

解析：HTTP/3 基于 QUIC，QUIC 运行在 UDP 之上。

## 13. 易错结论速背

- 三次握手确认双方收发能力和初始序列号。
- 四次挥手源于 TCP 全双工。
- TIME_WAIT 在主动关闭方，持续 2MSL。
- CLOSE_WAIT 在被动关闭方，大量出现多半是应用没 close。
- `select` 有 `FD_SETSIZE` 限制，且会修改 fd_set。
- `poll` 无固定 1024 限制，但仍线性扫描。
- `epoll` 用红黑树维护关注集合，用就绪链表保存 ready fd。
- LT 是水平触发，ET 是边沿触发。
- ET 要配合非阻塞 fd，并读写到 `EAGAIN`。
- Reactor 是就绪通知，Proactor 是完成通知。
- 惊群是多个等待者被无效唤醒。
- `SO_REUSEPORT` 可让多个 socket 绑定同一端口并负载均衡。
- HTTP/1.1 默认长连接；HTTP/2 多路复用但仍基于 TCP；HTTP/3 基于 QUIC/UDP。
- HTTPS = HTTP + TLS，提供加密、完整性、身份认证。
